## 0. 目标与边界

M3.0 建立 M3.1 和棋规则、M3.2 置换表共同依赖的状态基础：

1. 为每个 `Position` 维护确定性的 64 位 Zobrist key。
2. `make_move` 增量更新 key，`unmake_move` 精确恢复旧 key。
3. 引入 `GameState`，同时保存当前 `Position` 和真实对局历史 key。
4. UCI `position ... moves ...` 不再丢弃历史。
5. 搜索维护独立的当前搜索路径 key stack。
6. 现有 halfmove clock 随根局面进入搜索，并在搜索走子/撤销时正确更新和恢复。
7. 为后续三次重复、五十回合与 TT 提供稳定、测试充分的接口。

### 三条最高优先级不变式

1. **任何时刻：**

   ```rust
   pos.zobrist_key() == recompute_zobrist(pos)
   ```

2. **任何合法走子的 make/unmake 往返后：**

   ```text
   Position 全字段、FEN、Zobrist key 全部恢复到原值
   ```

3. **M3.0 不改变棋力行为：**

   本阶段不执行重复判和、五十回合判和或 TT 查询，因此固定深度搜索的 score、bestmove、PV 和节点数必须与 M2.4 完全一致。

## 1. 非目标

M3.0 明确不实现：

* 三次重复判和；
* 五十回合判和；
* insufficient material；
* Transposition Table；
* Exact / Lower / Upper bound；
* hash move ordering；
* `setoption name Hash`；
* `ucinewgame` 清 TT；
* mate score 的 TT ply 编解码；
* killer / history；
* aspiration / PVS；
* 任何 move ordering、alpha-beta 或评估修改；
* 新的 UCI 可见输出；
* 运行时随机数或外部随机数 crate。

M3.0 中历史和 hash stack **只维护，不读取并改变搜索结果**。

## 2. 状态分层

### 2.1 `Position`

`Position` 继续代表一个完整、可独立搜索的棋盘状态，包括现有的：

* board；
* side to move；
* castling rights；
* en-passant target；
* halfmove clock；
* fullmove number；
* king squares。

M3.0 新增当前局面的派生字段。该字段为 **crate-private 可写、公开只读**，通过方法暴露：

```rust
pub type ZobristKey = u64;

#[derive(Clone, Copy, Debug)]
pub struct Position {
    pub(crate) board: [Option<Piece>; 64],
    pub(crate) side: Color,
    pub(crate) castling: CastlingRights,
    pub(crate) ep_target: Option<Square>,
    pub(crate) halfmove: u32,
    pub(crate) fullmove: u32,
    pub(crate) king_sq: [Square; 2],
    pub(crate) zobrist_key: ZobristKey,
}

impl Position {
    pub fn board(&self) -> &[Option<Piece>; 64];
    pub fn side_to_move(&self) -> Color;
    pub fn castling_rights(&self) -> CastlingRights;
    pub fn ep_target(&self) -> Option<Square>;
    pub fn halfmove_clock(&self) -> u32;
    pub fn fullmove_number(&self) -> u32;
    pub fn king_square(&self, color: Color) -> Square;
    pub fn zobrist_key(&self) -> ZobristKey;
}
```

要求：

* 不把任何字段设为 public writable；`Position` 所有字段统一为 `pub(crate)`（见上方 struct 定义）；
* 不提供任何公开 mutable getter；外部修改局面只能通过合法构造（`parse_fen` / `startpos`）、`make_move` 和 `unmake_move` API；
* crate 内部模块（`fen`、`zobrist`、`search` 等）仍可直接读取 / 维护字段；integration tests 改用对应 getter；
* `Position` 继续保持 `Clone + Copy`；
* 这属于 0.x 阶段接受的 API tightening：新增私有 `zobrist_key` 字段本来就已经阻止外部 struct literal，因此此处不再保留一个会破坏 hash 的半公开状态；
* 外部读取字段一律通过对应 getter：`board()`、`side_to_move()`、`castling_rights()`、`ep_target()`、`halfmove_clock()`、`fullmove_number()`、`king_square()`、`zobrist_key()`。

### 2.2 `GameState`

新增：

```rust
#[derive(Clone)]
pub struct GameState {
    position: Position,
    key_history: Vec<ZobristKey>,
}
```

其不变式是：

```rust
!key_history.is_empty()
key_history.last() == Some(&position.zobrist_key())
```

`key_history` 包含：

```text
基础局面 key
应用第 1 个 UCI move 后的 key
应用第 2 个 UCI move 后的 key
...
当前局面 key
```

例如：

```text
position startpos moves e2e4 e7e5 g1f3
```

历史长度必须为 4：

```text
[startpos, after e2e4, after e7e5, after g1f3]
```

### 2.3 搜索路径状态

搜索不能直接修改 `GameState.key_history`。

搜索内部新增私有结构：

```rust
struct SearchPath {
    keys: Vec<ZobristKey>,
    root_len: usize,
}
```

开始搜索时：

```text
SearchPath.keys = GameState.key_history 的副本
root_len = keys.len()
```

搜索每 make 一个走子后 push 新 key，unmake 时 pop。

任何正常返回、中止返回、beta cutoff、qsearch cutoff 和 emergency evasion 返回后，都必须恢复：

```rust
path.keys.len() == path.root_len
path.keys.last() == Some(&root_position.zobrist_key())
```

## 4. 确定性 Zobrist 表

### 4.1 禁止运行时随机化

Zobrist key 必须满足：

* 所有机器一致；
* debug / release 一致；
* 每次启动一致；
* 不依赖系统时间；
* 不依赖 `rand` crate；
* 可锁定固定测试常量。

使用固定 seed 和 SplitMix64 的 compile-time 表生成。

### 4.2 固定 seed 与 slot 映射

锁定：

```rust
const ZOBRIST_SEED: u64 = 0xD1B5_4A32_D192_ED03;
const SPLITMIX_GAMMA: u64 = 0x9E37_79B9_7F4A_7C15;
```

每个 key 由唯一 slot 生成：

```rust
const fn zobrist_value(slot: u64) -> u64 {
    let mut z = ZOBRIST_SEED
        .wrapping_add((slot + 1).wrapping_mul(SPLITMIX_GAMMA));

    z = (z ^ (z >> 30)).wrapping_mul(0xBF58_476D_1CE4_E5B9);
    z = (z ^ (z >> 27)).wrapping_mul(0x94D0_49BB_1331_11EB);
    z ^ (z >> 31)
}
```

slot 分配严格锁定为：

```text
0..=767     piece-square keys
768         black-to-move key
769..=784   castling mask 0..=15
785..=792   en-passant file a..h
```

总 slot 数：

```text
793
```

> **固定常量不得改动：** 上述 `ZOBRIST_SEED`、`SPLITMIX_GAMMA`、slot 映射、startpos 固定结果（`0x61A2_181F_8F2F_9B9C`，见 §10.1）均为锁定值。若实现后实测结果不同，必须回溯检查 piece/color 映射、square 方向、castling mask、side key、SplitMix64 slot 公式，并修正映射/seed/公式本身——**不得修改 seed、mapping 或测试常量来迎合错误实现。**

### 4.3 Piece slot

颜色映射：

```text
White = 0
Black = 1
```

棋子映射：

```text
Pawn   = 0
Knight = 1
Bishop = 2
Rook   = 3
Queen  = 4
King   = 5
```

必须使用穷尽 `match`，不能依赖 enum 的 `as usize` 排列：

```rust
fn color_index(color: Color) -> usize {
    match color {
        Color::White => 0,
        Color::Black => 1,
    }
}

fn piece_index(piece_type: PieceType) -> usize {
    match piece_type {
        PieceType::Pawn => 0,
        PieceType::Knight => 1,
        PieceType::Bishop => 2,
        PieceType::Rook => 3,
        PieceType::Queen => 4,
        PieceType::King => 5,
    }
}
```

Piece slot：

```rust
((color_index * 6 + piece_index) * 64 + square) as u64
```

### 4.4 Castling mask

锁定 bit：

```text
White kingside  = 1
White queenside = 2
Black kingside  = 4
Black queenside = 8
```

所有局面都 XOR 对应的：

```rust
CASTLING_KEYS[mask]
```

包括 `mask == 0`。

### 4.5 Side key

约定：

```text
White to move：不 XOR SIDE_KEY
Black to move：XOR SIDE_KEY
```

## 5. Zobrist identity 包含什么

Zobrist key 包含：

* 每个棋子的颜色、类型和 square；
* side to move；
* 完整 castling-rights mask；
* 有效 en-passant file。

Zobrist key 不包含：

* halfmove clock；
* fullmove number；
* UCI 历史长度；
* 搜索深度；
* 评估值；
* check 状态缓存。

因此两个 FEN 若仅 halfmove / fullmove 数字不同，其 Zobrist key 必须相同。

### 5.1 为什么不 hash clocks

三次重复判断关注的是棋盘、走子方、王车易位权和实际可用的 en-passant 权。

halfmove / fullmove 是规则上下文，不是棋盘合法着法身份的一部分。

M3.2 使用 TT 时必须单独解决 rule-50 context：

> 不得因为 Zobrist key 相同，就在 halfmove clock 不同的局面间盲目复用与五十回合有关的 exact score。

该问题留给 M3.2 spec，不在 M3.0 偷做 TT。

## 6. En-passant hash 语义

### 6.1 只 hash “实际合法可吃”的 EP file

不能简单地对所有非空 `ep_target` 加 hash。

以下局面即使 FEN 有 EP target，也不应改变 repetition key：

1. 走子方没有相邻兵；
2. 相邻棋子不是本方兵；
3. 被吃位置没有敌方兵；
4. en-passant 会让本方王暴露在将军下。

因此：

```rust
effective_ep_file(pos) -> Option<u8>
```

只有在当前走子方至少存在一个**合法** en-passant capture 时才返回 file。

### 6.2 禁止通过完整 movegen 形成初始化递归

FEN 初始化需要调用 `recompute_zobrist`；若 `effective_ep_file` 再调用依赖有效 Zobrist key 的完整 `make_move`，容易形成循环依赖。

推荐使用专用只读合法性探测：

```rust
fn effective_ep_file(pos: &Position) -> Option<u8> {
    let target = pos.ep_target?;

    // 前置条件：EP target 必须为空。若被占用，该 FEN 本身非法
    // （见 §10 的 parse_fen 报错约束），此处直接返回 None。
    if pos.board[target as usize].is_some() {
        return None;
    }

    // EP target 必须位于本方兵能到达的 rank：
    // 白兵前进到 rank 6（index 5），黑兵前进到 rank 3（index 2）。
    let expected_rank = match pos.side {
        Color::White => 5,
        Color::Black => 2,
    };
    if rank_of(target) != expected_rank {
        return None;
    }

    let us = pos.side;
    let them = us.opposite();

    // 找 target 两侧可能进行 EP 的本方兵。
    // 验证被吃 square 上确实是敌方 Pawn。
    // 对每个候选：
    //   clone Position；
    //   手工在 clone.board 上移走 from pawn；
    //   移除被吃 pawn；
    //   把本方 pawn 放到 target；
    //   king_sq 不变；
    //   检查本方王是否被 them 攻击。
    //
    // 任一候选合法则返回 target file。
}
```

该探测：

* 不修改原 Position；
* 不调用递归 hash 初始化；
* 能识别 horizontal pin 等 en-passant 特殊自将情况；
* 只在 `ep_target.is_some()` 时执行。

**临时副本规则（明确）：** 为检查 pinned en-passant 而手工修改的 Position 副本，其 `zobrist_key` 在该局部会暂时与 board 不一致——这是允许的局部实现细节。但该临时副本**只能**用于：

* 手工移动 / 移除 EP 相关棋子（不改调用 `make_move`）；
* 调用 `is_square_attacked()`；
* 检查本方 King 是否暴露。

该路径**禁止**调用：

* `make_move()`；
* `generate_legal_moves()`；
* `recompute_zobrist()`；
* 任何读取临时副本 `zobrist_key` 的逻辑。

函数退出前不需要修复临时副本（它立即被丢弃）；原始 `Position` 绝不能被修改。

### 6.3 必测五种 EP

1. **合法 EP：key 应不同**

   ```text
   4k3/8/8/3pP3/8/8/8/4K3 w - d6 0 1
   ```

   与同局面 `ep_target = -` 的 key 不同。

2. **无相邻兵：key 应相同**

   ```text
   4k3/8/8/8/4P3/8/8/4K3 b - e3 0 1
   ```

   与 `ep_target = -` 的 key 相同。

3. **EP 因自将非法：key 应相同**

   ```text
   4r1k1/8/8/3pP3/8/8/8/4K3 w - d6 0 1
   ```

   白兵 `e5xd6 e.p.` 会打开黑车 e8 到白王 e1 的 e-file，因此 EP 不合法；它与 `ep_target = -` 的 key 必须相同。

4. **EP target 被占用：parse_fen 必须报错**

   ```text
   4k3/8/3n4/3pP3/8/8/8/4K3 w - d6 0 1
   ```

   d6 已被黑 Knight 占用，`parse_fen` 必须返回 `Err("en passant target square must be empty")`。

5. **被吃格无敌方兵：key 应相同**

   ```text
   4k3/8/8/8/4P3/8/8/4K3 w - d6 0 1
   ```

   白兵 e5 与 EP target d6 同处合法 rank，但 d5 没有敌方兵可被吃，因此 `effective_ep_file` 返回 `None`；该局面有 / 无 EP target 的 key 必须相同（normalized-hash 测试）。

## 7. 完整重算函数

在 `zobrist.rs` 中提供唯一的参考实现：

```rust
pub fn recompute_zobrist(pos: &Position) -> ZobristKey
```

流程：

```rust
let mut key = 0;

for square in 0..64 {
    if let Some(piece) = pos.board[square] {
        key ^= piece_square_key(piece, square);
    }
}

if pos.side == Color::Black {
    key ^= SIDE_KEY;
}

key ^= CASTLING_KEYS[castling_mask(pos.castling)];

if let Some(file) = effective_ep_file(pos) {
    key ^= EP_FILE_KEYS[file as usize];
}

key
```

用途：

* FEN / startpos 初始化；
* 测试增量 key；
* debug 工具。

禁止在正常搜索的每次 `make_move` 后调用 64-square 全量重算。

## 8. `make_move` 增量更新

### 8.1 Undo 扩展

现有 `Undo` 增加：

```rust
previous_zobrist_key: ZobristKey
```

如实现中缓存 effective EP file，也必须存入 Undo 并恢复。

### 8.2 通用更新顺序

`make_move` 开始时保存：

```rust
let previous_zobrist_key = self.zobrist_key;
let old_ep_file = effective_ep_file(self);
let old_castling_mask = castling_mask(self.castling);
```

从 key 中移除旧状态：

```rust
if let Some(file) = old_ep_file {
    self.zobrist_key ^= EP_FILE_KEYS[file as usize];
}

self.zobrist_key ^= CASTLING_KEYS[old_castling_mask];
```

然后根据走子类型更新棋子 key 和 board。

更新完 board、castling、EP、clocks 和 side 后：

```rust
self.zobrist_key ^= SIDE_KEY;
self.zobrist_key ^= CASTLING_KEYS[new_castling_mask];

if let Some(file) = effective_ep_file(self) {
    self.zobrist_key ^= EP_FILE_KEYS[file as usize];
}
```

### 8.3 普通走子

```text
XOR moving piece @ from
XOR moving piece @ to
```

若有 capture：

```text
XOR captured piece @ to
```

### 8.4 Promotion

```text
XOR Pawn @ from
XOR captured piece @ to（若有）
XOR promoted piece @ to
```

不能 XOR Pawn @ to。

### 8.5 En passant

```text
XOR moving Pawn @ from
XOR moving Pawn @ target
XOR captured enemy Pawn @ actual captured square
```

不能按普通 capture 在 target 上找被吃子。

### 8.6 Castling

除 King 外还必须更新 Rook：

```text
XOR King old/new
XOR Rook old/new
```

同时 castling mask 会从旧值更新为新值。

### 8.7 Castling-rights 变化

以下都必须反映到新 mask：

* King move；
* Rook 从原始角移动；
* 原始角上的 Rook 被吃。

不能只根据 move flag 判断。

## 9. `unmake_move`

`unmake_move` 继续按现有逻辑恢复：

* board；
* captured piece；
* promotion；
* en passant；
* rook castling move；
* side；
* castling；
* EP target；
* halfmove；
* fullmove；
* king squares。

Zobrist 不采用反向重演一套复杂 XOR。

完成字段恢复后直接：

```rust
self.zobrist_key = undo.previous_zobrist_key;
```

这样可以避免 make / unmake 两套 hash 逻辑逐渐漂移。

## 10. FEN 与 startpos 初始化

FEN parser 必须先完成并验证所有原有字段，包括：

* board；
* 双王；
* side；
* castling；
* EP；
* halfmove；
* fullmove；
* king squares。

EP target 合法性必须额外强制：若 FEN 给出的 `ep_target` 指向 square 上**已有棋子**（即 `board[ep_target].is_some()`），`parse_fen` 必须返回 `Err("en passant target square must be empty")`。该约束与 `effective_ep_file` 的前置条件一致——被占用的 EP target 不是合法局面。

然后最后执行：

```rust
pos.zobrist_key = recompute_zobrist(&pos);
```

`Position::startpos()` 必须走同一 hash 语义。

### 10.1 固定 startpos key

按本 spec 的 seed、slot 分配、piece mapping、side 约定和 castling mask，startpos key 锁定为：

```rust
0x61A2_181F_8F2F_9B9C
```

测试必须精确断言该值。

若实现结果不同，不能修改测试迎合实现；必须检查：

* piece / color slot 映射；
* square 方向；
* castling mask；
* side key；
* SplitMix64 slot 生成公式。

## 11. `GameState` API

推荐接口（公开只读、crate-private 可写）：

```rust
pub enum GameStateError {
    /// 传入的 Move 不在当前局面的合法着法列表中。
    MoveNotLegal(Move),
}

impl GameState {
    pub fn startpos() -> Self;

    pub fn from_position(position: Position) -> Self;

    pub fn position(&self) -> &Position;

    pub fn key_history(&self) -> &[ZobristKey];

    pub fn current_key(&self) -> ZobristKey;

    /// 在 Position 的副本上生成合法着法，不暴露 &mut Position。
    pub fn legal_moves(&self) -> Vec<Move>;

    /// 验证 mv 存在于合法着法列表；失败则 GameState 完全不变。
    pub fn apply_legal_move(&mut self, mv: Move) -> Result<(), GameStateError>;

    /// 仅由已完成合法性验证的 crate 内部路径调用。
    pub(crate) fn push_known_legal_move(&mut self, mv: Move);

    pub(crate) fn into_search_parts(self) -> (Position, Vec<ZobristKey>);
}
```

契约：

* `legal_moves()` 在 `Position` 的副本上执行 move generation，不暴露 `&mut Position`；
* `apply_legal_move()` 必须验证传入的完整 `Move` 存在于合法着法列表中，失败时 `GameState` 完全不变（返回 `Err(GameStateError::MoveNotLegal(mv))`）；
* `push_known_legal_move()` 只能由已经完成合法性验证的 crate 内部路径调用，调用方必须保证 `mv` 合法；
* 所有成功走子必须同步更新 `Position` 和 `key_history`；
* `key_history.last() == position.zobrist_key()` 永远成立。

`from_position` 必须初始化（先校验派生状态一致，再建 history）：

```rust
debug_assert_eq!(
    position.zobrist_key(),
    recompute_zobrist(&position),
);
key_history = vec![position.zobrist_key()]
```

`push_known_legal_move` 实现为：

```rust
self.position.make_move(mv);
self.key_history.push(self.position.zobrist_key());
```

`apply_legal_move` 实现为：生成 `legal_moves()`，若 `mv` 不在其中则返回 `Err`，否则调用 `push_known_legal_move(mv)`。

**不要提供 `position_mut_for_movegen()`** —— 它会允许调用方修改 `Position` 却不更新 `key_history`，导致二者脱同步。

**不要公开一个假定调用者已验证合法性的 `push_legal_move()`** —— 公开方法必须验证；已知合法的快速追加只能是 crate-private 的 `push_known_legal_move`。

## 12. UCI 集成

### 12.1 主循环状态

把：

```rust
let mut pos = Position::startpos();
```

替换为：

```rust
let mut game = GameState::startpos();
```

### 12.2 `position` 命令保持原子性

推荐把现有“原地修改 Position”的接口改成：

```rust
fn parse_position_command(tokens: &[&str]) -> Result<GameState, String>
```

`find_move` 改为接收只读 `Position`（不得为了复用现有 `find_move(&mut Position, ...)` 而重新向 `GameState` 暴露 `&mut Position`）：

```rust
pub fn find_move(pos: &Position, uci: &str) -> Option<Move> {
    let mut probe = *pos;
    let moves = generate_legal_moves(&mut probe);
    // 匹配完整 from / to / promotion。
}
```

处理步骤：

1. 从 `startpos` 或 FEN 构造临时 `GameState`；
2. 对 `moves` 后每个 token：

   * 在临时 position 上（通过 `find_move(&position, token)`）找到完整 flag / promotion 的 `Move`；
   * `apply_legal_move(mv)`；
3. 所有 token 成功应用后返回新 `GameState`；
4. 任一 move 非法或 `apply_legal_move` 返回 `Err`，则原主循环 `game` 完全不变（临时 `GameState` 被丢弃），返回错误。

主循环：

```rust
match parse_position_command(&tokens) {
    Ok(new_game) => game = new_game,
    Err(e) => println!("info string {}", e),
}
```

### 12.3 FEN 历史边界

对：

```text
position fen <FEN>
```

引擎无法推断 FEN 之前的真实对局历史，因此：

```text
history = [FEN root key]
```

若命令继续带 moves，则从该 FEN 开始追加。

不得伪造 FEN 之前的重复记录。

### 12.4 `ucinewgame`

本阶段执行：

```rust
game = GameState::startpos();
```

没有 TT，因此不做 TT clear。

### 12.5 搜索线程

搜索线程接收一个 `GameState` clone，而不是只接收 `Position`：

```rust
fn spawn_search(
    game: GameState,
    limits: SearchLimits,
    stop: Arc<AtomicBool>,
    budget: TimeBudget,
) -> JoinHandle<()>
```

在线程内拆出：

```rust
let (mut pos, game_history) = game.into_search_parts();
```

并调用 history-aware 私有搜索入口。

主线程中的 `game` 不被搜索修改。

### 12.6 Perft

`perft` 只针对当前 position：

```rust
let mut pos = game.position().clone();
let nodes = pos.perft(depth);
```

perft 不修改真实 `GameState` 或 `key_history`。

## 13. 搜索 API 与路径接线

### 13.1 公共签名保持不变

现有公开 API 必须原样保留：

```rust

pub fn quiescence(...) -> Option<i32>;
pub fn search_best_move(...) -> Option<SearchOutcome>;
```

现有集成测试和外部调用方必须继续编译。

这些兼容包装器使用：

```text
只有当前根 key 的单元素 history
```

即它们支持搜索路径状态，但不知道根局面以前的真实对局历史。

### 13.2 新增 crate-private history-aware 入口

```rust
pub(crate) fn search_best_move_with_history(
    pos: &mut Position,
    game_history: &[ZobristKey],
    limits: &SearchLimits,
    ctx: &SearchContext,
) -> Option<SearchOutcome>;
```

契约：

```rust
!game_history.is_empty()
game_history.last() == Some(&pos.zobrist_key())
```

实现入口处必须额外执行派生状态不变式检查：

```rust
debug_assert_eq!(
    pos.zobrist_key(),
    recompute_zobrist(&pos),
);
```

该函数只由保持不变式的内部 `GameState` / UCI 路径调用。

### 13.3 私有递归函数增加 SearchPath

以下私有实现增加：

```rust
path: &mut SearchPath
```

包括：

* `negamax_impl`；
* `quiescence_impl`；
* `quiescence_entered_impl`；
* `search_final_evasion_ply`；
* `root_search`。

每条子线统一：

```rust
let undo = pos.make_move(mv);
path.push(pos.zobrist_key());

let child = recurse(...);

pos.unmake_move(undo);
```

然后才处理 child 的 `Some` / `None`。

所有分支都必须成对执行 push / pop 与 make / unmake，包括：

* 正常 alpha improvement；
* beta cutoff；
* qsearch cutoff；
* node-budget abort；
* stop abort；
* hard-deadline abort；
* emergency final evasion；
* terminal child。

### 13.4 暂不判和

M3.0 中不能出现：

```rust
if repeated { return Some(0); }
if halfmove >= 100 { return Some(0); }
```

SearchPath 在本阶段仅维护数据。

## 14. Halfmove clock

halfmove clock 继续属于 `Position`，不复制进：

* `SearchContext`；
* `SearchLimits`；
* `GameState` 独立字段；
* `SearchPath`。

搜索通过自身正在 make / unmake 的 `Position` 读取当前 halfmove clock。

必须锁定：

* Pawn move → 0；
* normal capture → 0；
* en-passant → 0；
* promotion → 0；
* quiet non-pawn move → `old + 1`；
* castling → `old + 1`；
* unmake → 精确恢复旧值。

fullmove number 继续按现有规则维护，但不进入 Zobrist key。

---

## 15. 热路径约束

* 正常 `make_move` 不得扫描全部 64 squares 重算 hash；
* 不引入运行时随机初始化；
* 不在每个节点 clone 整个 GameState；
* 搜索线程启动时只 clone 一次 UCI history；
* SearchPath 在搜索开始时 reserve 足够空间；
* 不根据用户传入的荒谬 depth 一次性分配同等大小；
* effective EP 的局部 Position clone 只在 `ep_target.is_some()` 时允许；
* hash 维护不得触发额外搜索节点；
* 不改变 move generation 次序或 alpha-beta 分支。

## 16. 必须测试

### 16.1 确定性与初始化

1. startpos：

   ```rust
   assert_eq!(
       Position::startpos().zobrist_key(),
       0x61A2_181F_8F2F_9B9C
   );
   ```

2. 同一 FEN 重复解析得到相同 key。

3. `pos.zobrist_key() == recompute_zobrist(&pos)`。

4. debug / release 使用同一固定值。

### 16.2 Hash component

分别构造只改变以下单项的两个局面，key 必须不同：

* side to move；
* 任一棋子 square；
* 棋子类型；
* 棋子颜色；
* castling rights；
* 合法 en-passant file。

仅改变以下字段，key 必须相同：

* halfmove clock；
* fullmove number；
* 无法实际执行的 EP target。

### 16.3 En-passant

必须覆盖 §6.3 五种局面：

* 合法 EP → 有 / 无 target key 不同；
* 无相邻兵 → key 相同；
* pinned EP → key 相同；
* EP target 被占用 → `parse_fen` 返回 `Err("en passant target square must be empty")`；
* 被吃格无敌方兵 → 有 / 无 target key 相同（normalized-hash）。

### 16.4 Make / unmake 特殊走子

对以下类型分别断言：

```text
make 后 incremental == recompute
unmake 后 FEN == before
unmake 后 key == before key
```

覆盖：

* quiet move；
* capture；
* double pawn push；
* en-passant；
* white kingside castle；
* black queenside castle；
* quiet promotion；
* capture promotion；
* King move 失去 castling；
* Rook move 失去 castling；
* 原始角 Rook 被吃失去 castling。

### 16.5 多层递归 hash walk

新增测试 helper，对选定 FEN 做浅层递归：

```rust
fn hash_walk(pos: &mut Position, depth: u32) {
    assert_eq!(pos.zobrist_key(), recompute_zobrist(pos));

    if depth == 0 {
        return;
    }

    for mv in generate_legal_moves(pos) {
        let before_fen = to_fen(pos);
        let before_key = pos.zobrist_key();

        let undo = pos.make_move(mv);
        assert_eq!(pos.zobrist_key(), recompute_zobrist(pos));

        hash_walk(pos, depth - 1);

        pos.unmake_move(undo);
        assert_eq!(to_fen(pos), before_fen);
        assert_eq!(pos.zobrist_key(), before_key);
    }
}
```

至少运行：

* startpos depth 3；
* 含 castling 的局面 depth 2；
* 含 EP 的局面 depth 2；
* 含 promotion 的局面 depth 2。

### 16.6 Transposition

从 startpos 使用两种走子顺序：

```text
A: g1f3 g8f6 b1c3 b8c6
B: b1c3 b8c6 g1f3 g8f6
```

断言：

* 最终 FEN 相同；
* 最终 Zobrist key 相同；
* 两个 GameState 的历史序列不同；
* current key 相同。

### 16.7 GameState

2. N 个 move 后 len == N + 1；

## 20. 最终锁死

M3.0 最重要的五点：

1. Zobrist 算法、seed、slot mapping 和 startpos key 固定，不能实现后修改测试迎合。
2. EP 只在存在合法 en-passant capture 时进入 hash，包括 pinned-EP 判别。
3. `make_move` 增量更新，`unmake_move` 从 Undo 精确恢复旧 key。
4. GameState 保存真实 UCI 历史，搜索使用独立 path stack，二者不互相污染。
5. 本阶段只维护状态，不判和、不查 TT；M2.4 的 nodes、score、bestmove 和 PV 全部保持不变。

> M3.0 涉及 Position 核心状态、所有 make / unmake、UCI 生命周期和整棵搜索树，比 M2.x 风险高很多。当前 Position 是 Copy 且所有状态字段公开，Undo 负责完整恢复；UCI 的 position 命令也已经通过临时 new_pos 实现原子替换。新设计必须贴合这些真实结构，不能边实现边发明接口。
