# M3.1 Spec — Draw Rules (Threefold / Fifty-Move / Insufficient Material)

**Status:** DRAFT (spec-only; implementation per C1/C2/C3 after remote review)

**Target file:** `docs/specs/m3.1-draw-rules.md`

**Commit plan:**

```text
docs: specify M3.1 draw rules                  (this commit — spec only, then push)
feat(search): add insufficient-material draw      (C1, after review)
feat(search): add fifty-move draw              (C2, after review)
feat(search): add threefold-repetition draw     (C3, after review)
```

No version bump, no tag. Default-push applies (each commit is pushed after the
local gate passes, per the standing 2026-07-19 discipline).

M3.1 adds the three draw predicates and wires them into the search so that a
drawn node scores `0`. It deliberately does NOT implement the Transposition
Table (that is M3.2), and it does NOT change `evaluate`, move ordering,
alpha-beta, or any public search-API signature.

---

## 0. Goals & boundaries

M3.1 makes the engine treat three classes of drawn positions as a `0` score
inside the search:

1. **Insufficient material** — K vs K, K+minor vs K, K+B vs K+B (bishops
   on the same color square).
2. **Fifty-move rule** — `halfmove_clock >= 100`.
3. **Threefold repetition** — a position that has already appeared earlier in
   the combined game-history + search line.

### Three invariants that must hold

1. **Zobrist identity unchanged** (M3.0 §16.1): at all times
   `pos.zobrist_key() == recompute_zobrist(pos)`.
2. **M3.0 §16.9 behavior baselines are UNCHANGED**: for the two locked
   positions — startpos depth-3 (`1149` / `b1c3` / `cp 50` / `pv
   [b1c3,b8c6,g1f3]`) and queen-win depth-3 (`963` / `e4a4` / `cp 890`
   / `pv [e4a4,h4h3,a4h4,h8g8,h4h3]`) — `nodes`, `score`, `bestmove`,
   and `PV` are identical to M3.0. Neither position is a draw, so draw
   detection must not perturb them. This is the primary regression gate.
3. **SearchPath invariant unchanged** (M3.0 §16.8): every `make_move`
   still has exactly one `push_child` + one `pop`; draw checks READ `path`
   but never push/pop it, so `restore_root` behavior is unaffected.

## 1. Non-goals

M3.1 explicitly does NOT:

* implement the Transposition Table, Exact/Lower/Upper bounds, hash-move
  ordering, `setoption name Hash`, or `ucinewgame` TT clear (M3.2);
* implement the 75-move *mandatory* draw;
* add any new UCI command or visible output — a draw is reflected ONLY as a
  `0` score inside the search. No new `info` lines, no draw *claim* to UCI;
* bump the version number or create a tag;
* change `evaluate`, move ordering, the alpha-beta framework, or the public
  search-API signatures (`negamax`, `quiescence`, `search_best_move`,
  `search_best_move_with_history`);
* surface draw detection on the UCI side (e.g., reporting a draw via
  `info`) — deferred until needed.

## 2. Definitions

* **Repetition key** — the `ZobristKey` of a `Position`. It identifies a
  board position for repetition purposes. Per `zobrist.rs`, it deliberately
  EXCLUDES the halfmove clock and fullmove number, and includes the
  en-passant *file* only when a legal EP capture exists
  (`effective_ep_file`). Consequently two FENs that differ only in
  halfmove/fullmove hash identically — which is exactly what the FIDE
  threefold rule requires (halfmove/fullmove are not part of a "position").
* **Combined line** — inside the search, the relevant history is
  `SearchPath::keys()`, which is `[game_history…, search_ancestors…,
  current]`. The game-history prefix comes from `GameState::key_history()`
  (passed into `search_best_move_with_history` and cloned into the
  `SearchPath`).
* **Draw score** — `0` from the side-to-move's perspective. A drawn
  position is returned as `Some(0)` by `negamax_impl` /
  `search_final_evasion_ply`, and as `Some(RootIteration { score: 0, pv:
  vec![] })` by `root_search`.

## 3. Existing state the spec relies on (verified, do not re-derive)

* `Position` maintains `halfmove: u32` (getter `halfmove_clock()`) and
  `fullmove: u32` (getter `fullmove_number()`). `make_move` resets
  `halfmove` to `0` on a pawn move or a capture, otherwise `halfmove + 1`;
  `fullmove` is incremented only after Black's move. `unmake_move` restores
  both verbatim from `Undo`.
* `ZobristKey = u64` (see §2).
* `GameState::key_history() -> &[ZobristKey]` returns the real game history,
  ending with the current position's key. `GameState::startpos()` and
  `push_known_legal_move(Move)` build it incrementally.
* `SearchPath` (M3.0): `keys() -> &[ZobristKey]`; `push_child(&Position)`
  appends the CHILD key after each `make_move`; `pop()` pairs it after
  `unmake_move`. At a node, `path.keys().last() == pos.zobrist_key()`.
* `Piece { pub color: Color, pub piece_type: PieceType }`;
  `Position::board() -> &[Option<Piece>; 64]` (index = `rank*8 + file`,
  `a1 = 0`). `Square = u8`; `file_of(sq) = sq % 8`, `rank_of(sq) = sq / 8`.
  There is NO `SquareColor` enum — square color is computed as
  `(file_of(sq) + rank_of(sq)) % 2` (parity equality is what matters for
  "same color"; the absolute light/dark assignment is irrelevant).
* `PieceType` variants: `Pawn, Knight, Bishop, Rook, Queen, King`
  (`PieceType::value()` exists for material).
* Search return types: `negamax_impl(...) -> Option<i32>`; `SearchOutcome
  { best_move, score: Option<i32>, completed_depth, stopped, pv }`.
* `parse_fen` accepts the halfmove and fullmove fields (the START_FEN ends
  `0 1`), so FENs with a non-zero halfmove clock parse fine.

## 4. Predicates (`src/engine/draw.rs`, `pub(crate)`)

New module `draw.rs` holding the pure predicates plus the combined
classifier. It imports `use crate::chess::position::Position;`,
`use crate::chess::game::GameState;`, `use crate::chess::types::*;`, and
`use crate::engine::search::SearchPath;` (SearchPath is `pub(crate)`).

```rust
pub(crate) enum DrawReason {
    Threefold,
    FiftyMove,
    InsufficientMaterial,
}

/// Combined draw check for a search node. `path` is the live SearchPath
/// (read-only). Order: cheapest/most-determinate first.
pub(crate) fn classify_draw(pos: &Position, path: &SearchPath) -> Option<DrawReason> {
    if is_insufficient_material(pos) {
        return Some(DrawReason::InsufficientMaterial);
    }
    if is_fifty_move_draw(pos) {
        return Some(DrawReason::FiftyMove);
    }
    if is_repetition_draw(pos, path) {
        return Some(DrawReason::Threefold);
    }
    None
}

/// K vs K, K+minor vs K, K+B vs K+B (bishops same color). Winning
/// reductions (K+N+N vs K, K+B+N vs K) are NOT automatic draws.
pub(crate) fn is_insufficient_material(pos: &Position) -> bool {
    let mut bishops_light = 0u32;
    let mut bishops_dark = 0u32;
    let mut knights = 0u32;
    let mut kings = 0u32;
    let mut heavy_or_pawn = false;
    for sq in 0..64u8 {
        if let Some(p) = pos.board()[sq as usize] {
            match p.piece_type {
                PieceType::Pawn | PieceType::Rook | PieceType::Queen => {
                    heavy_or_pawn = true;
                }
                PieceType::King => kings += 1,
                PieceType::Knight => knights += 1,
                PieceType::Bishop => {
                    if (file_of(sq) + rank_of(sq)) % 2 == 0 {
                        bishops_light += 1;
                    } else {
                        bishops_dark += 1;
                    }
                }
            }
        }
    }
    if heavy_or_pawn {
        return false;
    }
    let minors = knights + bishops_light + bishops_dark;
    // K vs K
    if kings == 2 && minors == 0 {
        return true;
    }
    // K + single minor vs K  (KB vs K, KN vs K)
    if kings == 2 && minors == 1 {
        return true;
    }
    // K + B vs K + B, both bishops on the SAME color square.
    if kings == 2 && knights == 0 && (bishops_light + bishops_dark) == 2
        && (bishops_light == 2 || bishops_dark == 2)
    {
        return true;
    }
    false
}

/// Fifty-move rule: 100 half-moves (50 full moves) without a pawn move
/// or capture. (The 75-move mandatory draw is out of scope.)
pub(crate) fn is_fifty_move_draw(pos: &Position) -> bool {
    pos.halfmove_clock() >= 100
}

/// Repetition INSIDE the search line (see §6 for the 2-fold convention).
pub(crate) fn is_repetition_draw(pos: &Position, path: &SearchPath) -> bool {
    let key = pos.zobrist_key();
    // path.keys() == [game_history..., search_ancestors..., CURRENT].
    // Exclude the current node (the last element); count prior occurrences.
    let keys = path.keys();
    let prior = &keys[..keys.len().saturating_sub(1)];
    prior.iter().any(|&k| k == key)
}

/// Strict FIDE threefold against the actual played game (count >= 3 in the
/// game history). Provided for correctness of any future draw *claim* / UCI
/// reporting; NOT what the search uses node-by-node (see §6).
pub(crate) fn is_threefold_in_game(gs: &GameState) -> bool {
    let key = gs.current_key();
    gs.key_history().iter().filter(|&&k| k == key).count() >= 3
}
```

## 5. Search integration points

All three rules are consulted at node entry. Draw returns are `Some(0)`; no PV
row is written for a drawn node (the row was already cleared by
`pv.clear_at(ply)` / `PvTable` discipline).

* **`negamax_impl`** — after `try_enter_node(ctx, limits)` and
  `pv.clear_at(ply)`, BEFORE `generate_legal_moves` / the terminal check /
  the `depth == 0` hand-off:

  ```rust
  if let Some(_) = classify_draw(pos, path) {
      return Some(0);
  }
  ```

* **`root_search`** — at entry, before the root-move loop:

  ```rust
  if classify_draw(pos, path).is_some() {
      return Some(RootIteration { score: 0, pv: Vec::new() });
  }
  ```
  (`search_best_move_impl` still selects a legal `best_move` from
  `root_moves`, so UCI still emits a move; only the score is `0`.)

* **`search_final_evasion_ply`** — at entry, before the `legal` loop:

  ```rust
  if classify_draw(pos, path).is_some() {
      return Some(0);
  }
  ```

* **`quiescence_impl` / `quiescence_entered_impl`** — at entry, apply ONLY
  insufficient-material (no repetition, no fifty-move):

  ```rust
  if is_insufficient_material(pos) {
      return Some(0);
  }
  ```

  Rationale: quiescence only descends into captures, which RESET the halfmove
  clock, so a fifty-move draw cannot newly arise inside qsearch (and if it
  already held, the parent `negamax_impl` node returned `0` first); and a
  capture cannot reproduce a prior position, so a repetition cannot occur
  inside qsearch. Insufficient material CAN newly arise in qsearch (e.g., a
  capture removes the last non-king piece, leaving K vs K), so it must be
  checked.

The `SearchPath` is passed by `&` (read-only) into `classify_draw` /
`is_repetition_draw`; these functions never call `push_child` / `pop`. The
M3.0 §16.8 push/pop pairing is therefore untouched.

## 6. Repetition counting semantics (decision point)

**Search node:** we treat a position as a draw when its Zobrist key appears
EARLIER in the combined line (game-history prefix + search ancestors) — i.e.
the current node is at least the **2nd** occurrence. `is_repetition_draw`
returns `true` on that condition.

Rationale for the 2-fold-in-search convention (over a strict 3-fold count):

1. A player would claim the FIDE threefold on the third occurrence, so the
   position is *effectively* drawn the moment it would repeat. Scoring it `0`
   on the second occurrence reflects that reality.
2. Returning `0` on the second occurrence stops the engine from
   **repetition-seeking** (repeating to escape a worse position) and keeps
   reduced-material endgames (e.g. KQ vs K) terminating sensibly.

`is_threefold_in_game` (strict `count >= 3` in `key_history()`) is provided
separately for correctness of any future draw *claim* / UCI reporting. It is
NOT what the search consults per node.

**Optional future refinement (OUT of M3.1 scope):** use
`alpha = max(alpha, draw_score)` so the engine prefers to deviate from a
forced draw only when it can do better. Not required for M3.1.

## 7. Tests (acceptance gates)

### §N.1 `is_insufficient_material` (unit, known FENs)

* `8/8/8/8/8/8/k7/K7 w - - 0 1` → **true** (K vs K)
* `8/8/8/8/8/8/k7/KN6 w - - 0 1` → **true** (K+N vs K)
* `8/8/8/8/8/8/k7/KB6 w - - 0 1` → **true** (K+B vs K)
* `8/8/8/8/8/8/k1b5/K1B5 w - - 0 1` → **true** (K+B vs K+B, bishops
  on the same color square — verify by constructing both bishops on
  light-or-both-dark)
* `8/8/8/8/8/8/k1b5/K1b5 w - - 0 1` → **false** (bishops on opposite
  colors — NOT an automatic draw)
* `8/8/8/8/8/8/r7/K7 w - - 0 1` → **false** (rook)
* `8/8/8/8/8/8/p7/K7 w - - 0 1` → **false** (pawn)
* `8/8/8/8/8/8/q7/K7 w - - 0 1` → **false** (queen)
* `8/8/8/8/8/8/n7/KN6 w - - 0 1` → **false** (K+N+N vs K — not automatic)
* `8/8/8/8/8/8/b7/KN6 w - - 0 1` → **false** (K+B+N vs K — not automatic)

Draw is side-independent; assert with both `w` and `b` to move where easy.

### §N.2 `is_fifty_move_draw`

* `8/8/8/8/8/8/k7/K7 w - - 100 1` → **true**
* `8/8/8/8/8/8/k7/K7 w - - 99 1` → **false**

### §N.3 Threefold

* **(a) Game-level (strict):** build a `GameState` by replaying the knight
  shuffle `g1f3 g8f6 f3g1 f6g8 g1f3 g8f6 f3g1 f6g8` from startpos. After
  these 8 half-moves the position before White's next move equals startpos,
  so the startpos key occurs at ply 0, after move 4, and after move 8 →
  exactly **3 times**. Assert
  `gs.key_history().iter().filter(|&&k| k == startpos_key).count() == 3`
  and `is_threefold_in_game(&gs)` is **true**. (The startpos key constant is
  `0x61A2_181F_8F2F_9B9C` per `zobrist.rs`.)
* **(b) Search-level (white-box):** construct `let mut path =
  SearchPath::new(vec![key, key]);` for a normal (non-drawn-by-material,
  non-fifty-move) `pos` whose `zobrist_key() == key`, then call
  `negamax_impl(&mut pos, depth, 0, -INF, INF, &ctx, &limits, &mut pv,
  &mut path)`. Expect **`Some(0)`** — the 2nd occurrence in the line is a
  draw. (`negamax_impl` and `SearchPath::new` are reachable from the
  `search.rs` test module.)

### §N.4 Search integration returns `0` for a drawn node

* A K-vs-K position fed to `negamax_impl` (or `search_best_move`) returns
  score **`Some(0)`**.
* A root position whose FEN has `halfmove_clock >= 100` returns score
  **`Some(0)`** from `search_best_move` / `search_best_move_with_history`.

### §N.5 Regression — M3.0 §16.9 baselines must be UNCHANGED

* `tests/m3_0.rs` (the §16.9 behavior locks) still compiles and passes.
* Full `cargo test --release` still reports: startpos depth-3 =
  `1149` / `b1c3` / `cp 50` / `pv [b1c3,b8c6,g1f3]`; queen-win depth-3
  = `963` / `e4a4` / `cp 890` / `pv [e4a4,h4h3,a4h4,h8g8,h4h3]`.
* `perft_startpos_depth_5_release` still `ok` (= 4,865,609).
