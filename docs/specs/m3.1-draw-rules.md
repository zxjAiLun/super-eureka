## 0. Goals & boundaries

M3.1 makes the engine treat three classes of drawn positions correctly
inside the search:

1. **Insufficient material** — K vs K, K+minor vs K, K+B vs K+B (bishops
   on the same color square), and any number of same-color-square bishops
   vs K (dead position).
2. **Fifty-move rule** — `halfmove_clock >= 100`.
3. **Threefold repetition** — a position that has occurred at least **three**
   times in the combined game-history + search line.

### Two fundamentally different draw kinds (read this first)

These three rules are NOT the same kind of "draw", and conflating them is the
#1 source of wrong scores:

* **Automatic / dead-position draw** — `InsufficientMaterial`. The position
  is over by rule the instant it arises. It returns `0` **immediately** and
  terminates the search line. There is no option; no side can "win" from a
  dead position.
* **Claimable draw** — `FiftyMoveClaim` and `ThreefoldClaim`. Per FIDE
  these are claims *by the side to move*. They are a `0`-score **option**,
  not a mandatory terminal. If the side to move also has a winning line, the
  correct game-theoretic value is **at least `0`**, and the search must keep
  looking for that win. Forcing `0` unconditionally would throw away a mate.

Concretely, a claimable draw contributes a `0` **floor** to alpha and best,
and only allows a fail-hard cutoff when `0 >= beta`. It must NOT replace a
winning score with `0`.

### Layer-two: intended (prospective) claims (FIDE 9.2.1 / 9.3.1)

The `0` floor above is for a position that is **already** a claimable draw
(current node, current side to move). FIDE additionally lets the side to move
**declare an intended move before playing it**: if that move would produce the
third repetition or the 100th half-move, the player may claim on the move's
behalf *before* executing it. Concretely:

* if the PARENT is at `halfmove_clock == 99` and a quiet move makes the CHILD
  `halfmove_clock == 100`, the mover claims the fifty-move draw on that
  intended move and secures `0` — even if the resulting child position, had it
  been played without a claim, would be losing;
* if a move makes the child key the **third** occurrence in the line, the mover
  claims the threefold draw on that intended move and secures `0`.

This is NOT "the child side gets a `0` floor". The claim belongs to the
**mover of the parent edge**. Therefore, on every move edge, after
`make_move` + `push_child`, if the child is a claimable draw the move's score
is exactly `0` (unless the child is itself checkmate, which takes precedence
and keeps the win). See §5.1–§5.3 and the `claim_available_by_intended_move`
helper in §4.

### Three invariants that must hold

1. **Zobrist identity unchanged** (M3.0 §16.1): at all times
   `pos.zobrist_key() == recompute_zobrist(pos)`.
2. **M3.0 §16.9 behavior baselines are UNCHANGED**: for the two locked
   positions — startpos depth-3 (`1149` / `b1c3` / `cp 50` / `pv
   [b1c3,b8c6,g1f3]`) and queen-win depth-3 (`963` / `e4a4` / `cp 890`
   / `pv [e4a4,h4h3,a4h4,h8g8,h4h3]`) — `nodes`, `score`, `bestmove`,
   and `PV` are identical to M3.0. Neither position is a draw, so draw
   detection must not perturb them. This is the primary regression gate.
3. **SearchPath invariant unchanged** (M3.0 §16.8): every `make_move`
   still has exactly one `push_child` + one `pop`; draw checks READ `path`
   but never push/pop it, so `restore_root` behavior is unaffected.

## 2. Definitions

* **Repetition key** — the `ZobristKey` of a `Position`. It identifies a
  board position for repetition purposes. Per `zobrist.rs`, it deliberately
  EXCLUDES the halfmove clock and fullmove number, and includes the
  en-passant *file* only when a legal EP capture exists
  (`effective_ep_file`). Consequently two FENs that differ only in
  halfmove/fullmove hash identically — which is exactly what the FIDE
  threefold rule requires (halfmove/fullmove are not part of a "position").
* **Combined line** — inside the search, the relevant history is
  `SearchPath::keys()`, which is `[game_history…, search_ancestors…,
  current]`. The game-history prefix comes from `GameState::key_history()`
  (passed into `search_best_move_with_history` and cloned into the
  `SearchPath`).
* **Draw score** — for an **automatic** draw (insufficient material), the
  node returns `Some(0)` and the line ends. For a **claimable** draw
  (fifty-move / threefold), the node does NOT return `0` unconditionally;
  it applies a `0` floor to alpha and best (and may fail-hard cut off when
  `0 >= beta`), but continues searching for a better (winning) line. The
  eventual returned `Score` is therefore `max(search_result, 0)` from the
  side-to-move's perspective — never a forced `0` when a win exists.
* **RootIteration** (search-internal, changed by this milestone) — see §5.4.
  It carries an explicit `best_move: Move` so the root draw path never reads
  an (empty) PV to derive the move.

## 3. Existing state the spec relies on (verified, do not re-derive)

* `Position` maintains `halfmove: u32` (getter `halfmove_clock()`) and
  `fullmove: u32` (getter `fullmove_number()`). `make_move` resets
  `halfmove` to `0` on a pawn move or a capture, otherwise `halfmove + 1`;
  `fullmove` is incremented only after Black's move. `unmake_move` restores
  both verbatim from `Undo`.
* `Position::is_in_check(color: Color) -> bool`, `Position::is_checkmate()
  -> bool`, `Position::is_stalemate() -> bool`, and `Position::side`
  (the side to move) all exist and are used by the existing terminal check.
* `generate_legal_moves(&mut Position) -> Vec<Move>` is the existing call
  used at every node; it takes the **mutable** `pos` (already borrowed mutably
  after `make_move`). An empty result with `is_in_check(side)` true is
  checkmate, otherwise stalemate. (A read-only lookup — e.g. in UCI — copies
  the position first: `let mut probe = *pos; let moves =
  generate_legal_moves(&mut probe);`.) The move-edge pseudocode below passes
  `pos` directly because it is already `&mut Position`.
* `ZobristKey = u64` (see §2).
* `GameState::key_history() -> &[ZobristKey]` returns the real game history,
  ending with the current position's key. `GameState::startpos()` and
  `push_known_legal_move(Move)` build it incrementally.
* `SearchPath` (M3.0): `keys() -> &[ZobristKey]`; `push_child(&Position)`
  appends the CHILD key after each `make_move`; `pop()` pairs it after
  `unmake_move`. At a node, `path.keys().last() == pos.zobrist_key()`.
* `Piece { pub color: Color, pub piece_type: PieceType }`;
  `Position::board() -> &[Option<Piece>; 64]` (index = `rank*8 + file`,
  `a1 = 0`). `Square = u8`; `file_of(sq) = sq % 8`, `rank_of(sq) = sq / 8`.
  There is NO `SquareColor` enum — square color is computed as
  `(file_of(sq) + rank_of(sq)) % 2` (parity equality is what matters for
  "same color"; the absolute light/dark assignment is irrelevant).
* `PieceType` variants: `Pawn, Knight, Bishop, Rook, Queen, King`
  (`PieceType::value()` exists for material).
* Search return types: `negamax_impl(...) -> Option<i32>`; `SearchOutcome
  { best_move, score: Option<i32>, completed_depth, stopped, pv }`.
  `MATE = 1_000_000`. A checkmated node returns `Some(-(MATE - ply))`
  (negative, side-to-move's perspective); a stalemate returns `Some(0)`.
* `RootIteration` currently is `{ score: i32, pv: Vec<Move> }` (search.rs
  ~316). **This milestone changes it to carry `best_move` — see §5.4.**
* `parse_fen` accepts the halfmove and fullmove fields (the START_FEN ends
  `0 1`), so FENs with a non-zero halfmove clock parse fine.

## 4. Predicates (`src/engine/draw.rs`, `pub(crate)`)

New module `draw.rs` holding the pure predicates plus the combined
classifier. The pure draw module must NOT reverse-depend on `SearchPath`:
it receives the key slice `&[ZobristKey]` (the caller passes
`path.keys()`). The import list shown in the snippets below is the **union
across C1–C3**; each commit imports only what it actually uses (see §9):
C1 needs only `Position` (+ the `types::*` helpers for `PieceType`,
`file_of`, `rank_of`); `ZobristKey` is needed from C3 onward, and
`GameState` only in C3 (for `is_threefold_in_game`). Do NOT add unused
variants/imports in C1, and do NOT silence them with `allow(dead_code)` /
`allow(unused_imports)`.

```rust
/// Which kind of draw a node is.
pub(crate) enum DrawReason {
    /// Dead position: automatic, terminal, returns 0 immediately.
    InsufficientMaterial,
    /// FIDE fifty-move claim by the side to move: a 0-score OPTION,
    /// not a forced terminal.
    FiftyMoveClaim,
    /// FIDE threefold claim by the side to move: a 0-score OPTION,
    /// not a forced terminal.
    ThreefoldClaim,
}

/// Combined draw check for a search node. `keys` is the slice
/// `path.keys()` (read-only). Order: automatic / most-determinate first.
pub(crate) fn classify_draw(pos: &Position, keys: &[ZobristKey]) -> Option<DrawReason> {
    if is_insufficient_material(pos) {
        return Some(DrawReason::InsufficientMaterial);
    }
    if is_fifty_move_draw(pos) {
        return Some(DrawReason::FiftyMoveClaim);
    }
    if is_threefold_repetition(pos, keys) {
        return Some(DrawReason::ThreefoldClaim);
    }
    None
}

/// Current node, current side to move: can claim a draw RIGHT NOW
/// (FIDE 9.2.2 / 9.3.2). Underlying condition shared with
/// `claim_available_by_intended_move`. Used at a NODE ENTRY to give the
/// side to move a `0` floor.
pub(crate) fn claim_available_now(pos: &Position, keys: &[ZobristKey]) -> bool {
    is_fifty_move_draw(pos) || is_threefold_repetition(pos, keys)
}

/// A candidate move, once made, will produce a position the MOVER can claim
/// (FIDE 9.2.1 / 9.3.1: a player may declare an *intended* move and claim
/// before executing it). `child` is the position AFTER `make_move` +
/// `push_child`; `child_keys` is `path.keys()` at that point. This claim
/// belongs to the PARENT node's mover, NOT the child's side to move.
pub(crate) fn claim_available_by_intended_move(
    child: &Position,
    child_keys: &[ZobristKey],
) -> bool {
    is_fifty_move_draw(child) || is_threefold_repetition(child, child_keys)
}

/// K vs K, K+minor vs K, and ANY collection of same-color-square
/// bishops vs K — i.e. a dead position where no legal move sequence can
/// ever produce checkmate. Promotion can produce many same-color bishops;
/// they still cannot cover the other color, so they remain a dead position.
pub(crate) fn is_insufficient_material(pos: &Position) -> bool {
    let mut bishops_light = 0u32;
    let mut bishops_dark = 0u32;
    let mut knights = 0u32;
    let mut kings = 0u32;
    let mut heavy_or_pawn = false;
    for sq in 0..64u8 {
        if let Some(p) = pos.board()[sq as usize] {
            match p.piece_type {
                PieceType::Pawn | PieceType::Rook | PieceType::Queen => {
                    heavy_or_pawn = true;
                }
                PieceType::King => kings += 1,
                PieceType::Knight => knights += 1,
                PieceType::Bishop => {
                    if (file_of(sq) + rank_of(sq)) % 2 == 0 {
                        bishops_light += 1;
                    } else {
                        bishops_dark += 1;
                    }
                }
            }
        }
    }
    if heavy_or_pawn {
        return false;
    }
    let minors = knights + bishops_light + bishops_dark;
    // K vs K, or K + a single minor vs K (KB vs K, KN vs K).
    if kings == 2 && minors <= 1 {
        return true;
    }
    // Any number of bishops, no knights, all on ONE square color.
    // Covers K+B vs K+B and promotion-produced K+BB..B vs K.
    if kings == 2
        && knights == 0
        && (bishops_light + bishops_dark) > 0
        && (bishops_light == 0 || bishops_dark == 0)
    {
        return true;
    }
    false
}

/// Fifty-move rule: 100 half-moves (50 full moves) without a pawn move
/// or capture. (The 75-move mandatory draw is out of scope.)
pub(crate) fn is_fifty_move_draw(pos: &Position) -> bool {
    pos.halfmove_clock() >= 100
}

/// Strict FIDE threefold: the current position's key occurs at least THREE
/// times in `keys` (the combined game-history + search line).
pub(crate) fn is_threefold_repetition(pos: &Position, keys: &[ZobristKey]) -> bool {
    let key = pos.zobrist_key();
    keys.iter().filter(|&&k| k == key).count() >= 3
}

/// Strict FIDE threefold against the actual played game (count >= 3 in the
/// game history). Provided for correctness of any future draw *claim* / UCI
/// reporting; NOT what the search consults per node. Reuses the same slice
/// helper so there is only one repetition-counting semantics.
pub(crate) fn is_threefold_in_game(gs: &GameState) -> bool {
    let key = gs.current_key();
    gs.key_history().iter().filter(|&&k| k == key).count() >= 3
}
```

> **Twofold cycle cutoff (future, NOT M3.1):** if we ever want to prune on a
> *second* occurrence (a search heuristic), it must be a SEPARATE function
> with a distinct name (e.g. `is_twofold_cycle`) and a distinct
> `DrawReason` variant (e.g. `CycleCutoff`). It must NOT be called
> `Threefold`, must NOT reuse `is_threefold_repetition`, and must NOT be
> described as a FIDE draw rule — it is an optional search cutoff.

## 5. Search integration points

**Terminal check ALWAYS runs before any draw check.** Draw detection must
never mask a checkmate or stalemate. The order at every node is:

```text
try_enter_node
pv.clear_at(ply)
generate legal moves
if no legal moves:            # terminal — FIRST
    return mate_score or 0(stalemate)
inspect draw rules           # AFTER terminal
continue normal search
```

### 5.1 `negamax_impl` — every node

After `try_enter_node`, `pv.clear_at(ply)`, and the **terminal** check (mate
/ stalemate), inspect draws:

```rust
if let Some(reason) = classify_draw(pos, path.keys()) {
    match reason {
        DrawReason::InsufficientMaterial => return Some(0), // automatic
        DrawReason::FiftyMoveClaim | DrawReason::ThreefoldClaim => {
            // claimable: 0 is a floor, not a forced terminal
            if 0 >= beta {
                return Some(0); // fail-hard cutoff on the draw
            }
            alpha = alpha.max(0);
            best = best.max(0);
            // fall through to the normal move loop
        }
    }
}

// ... node-entry claim floor above; each move edge below respects the
// intended (prospective) claim (FIDE 9.2.1 / 9.3.1) ...
for mv in ordered_moves {
    let undo = pos.make_move(mv);
    path.push_child(pos);

    let child_moves = generate_legal_moves(pos);

    // Terminal (mate / stalemate) takes precedence over the intended claim.
    let mut score = if child_moves.is_empty() {
        terminal_child_score(pos) // mate: -(MATE-(ply+1)); stalemate: 0
    } else if claim_available_by_intended_move(pos, path.keys()) {
        // The mover can declare this intended move and claim before executing
        // it, securing at least 0. A non-terminal intended-claim move is
        // therefore scored EXACTLY 0 — NOT "the child side gets a floor".
        0
    } else {
        -negamax_impl(pos, depth - 1, ply + 1, -beta, -alpha, ctx, limits, pv, path)
    };

    path.pop();
    pos.unmake_move(undo);

    // ... existing best/alpha update + fail-high cutoff ...
}
```

Because the claimable branch falls through, a node that can claim a draw but
also has a winning move returns the winning score; a node with only
draw-or-worse moves returns `>= 0`. The `0 >= beta` cutoff is the only
place a claimable draw short-circuits.

### 5.2 `quiescence_entered_impl` — after terminal, check ALL rules

The previous assumption ("qsearch only descends into captures, so fifty-move
and repetition cannot arise") is WRONG: when in check, quiescence searches
*all* legal evasions, including quiet king moves and quiet blocks, which can
both advance `halfmove_clock` to 100 and repeat a prior position. Therefore
`quiescence_entered_impl` must check **all three** rules, with the same
automatic-vs-claimable split as §5.1:

```rust
// after the terminal (mate/stalemate) check inside quiescence_entered_impl
if let Some(reason) = classify_draw(pos, path.keys()) {
    match reason {
        DrawReason::InsufficientMaterial => return Some(0),
        DrawReason::FiftyMoveClaim | DrawReason::ThreefoldClaim => {
            if 0 >= beta { return Some(0); }
            // qsearch returns the max of the stand-pat score and the best
            // capture. A claimable draw contributes a 0 lower bound, so the
            // node value is `max(existing_qsearch_value, 0)` — never a forced
            // 0, never a forced loss. (Keep `alpha = alpha.max(0)` if the
            // entered variant still threads alpha.)
            let value = /* existing qsearch result at this node */ 0;
            return Some(value.max(0));
        }
    }
}

// Inside the qsearch MOVE LOOP (captures, and ALL evasions when in check),
// apply the SAME intended-claim rule as §5.1 so a prospective claim is not
// scored as a loss:
for mv in qmoves {
    let undo = pos.make_move(mv);
    path.push_child(pos);
    let child_moves = generate_legal_moves(pos);
    let score = if child_moves.is_empty() {
        terminal_child_score(pos) // mate / stalemate precedence
    } else if claim_available_by_intended_move(pos, path.keys()) {
        0 // mover can declare this intended (capture / evasion) move and claim
    } else {
        -qsearch_recurse(pos, ...)
    };
    path.pop();
    pos.unmake_move(undo);
    // ... qsearch max / alpha update ...
}
```

Check exactly ONCE, in `quiescence_entered_impl` (the entered variant). Do
NOT duplicate the check in the `quiescence` wrapper.

### 5.3 `search_final_evasion_ply` — check EACH child

The current entry-level `classify_draw` is insufficient: a quiet evasion can
push `halfmove_clock` to 100 *after* `make_move`, and that child must be
scored as a draw, not by the static capture approximation. Therefore:

1. At ply entry, the position itself may be an automatic draw — check
   `is_insufficient_material(pos)` and, if true, return `Some(0)`.
2. For each legal evasion `m`:
   - `make_move(m)`; `path.push_child(pos)`;
   - generate the CHILD's legal moves;
   - **terminal first**: if the child has no legal moves → it is mate or
     stalemate (score accordingly);
   - **then draw**: the evasion is an *intended move* the mover can declare
     and claim before executing it (FIDE 9.2.1 / 9.3.1), so:
     - `is_insufficient_material(&pos)` → child score `0` (automatic dead
       position — immediate terminal, not a claim);
     - `claim_available_by_intended_move(&pos, path.keys())` → child score
       `0` (the mover secures the draw on this intended evasion; the child
       cannot be a winning evasion here, so `0` is the correct value for the
       static-cap approximation);
   - otherwise use the existing static capture approximation;
   - `path.pop()`; `unmake_move(undo)`;

The key fix: the draw check runs on the **child** position (after push), not
only at ply entry.

### 5.4 `root_search` / `search_best_move_impl` — no empty-PV panic

`RootIteration` MUST carry the best move explicitly; the root draw path must
never read `pv[0]` (a drawn root can have an empty PV and would panic).

```rust
struct RootIteration {
    score: i32,
    best_move: Move,
    pv: Vec<Move>,
}
```

`search_best_move_impl` derives the move from `it.best_move`, NOT `it.pv[0]`
(currently at `completed.as_ref().map(|it| it.pv[0])` — this is the P0).
The root-move ordering block that currently reads `*m == pv[0]` must instead
use the local `best_move` already tracked in the loop.

Root draw handling (do this in `search_best_move_impl`, around the existing
depth loop — NOT by having `root_search` return an empty-PV `RootIteration`):

1. Generate the legal root moves first (as today).
2. If there are no legal moves, return the existing terminal
   (`Some(mate)` / `Some(0)` stalemate) — unchanged.
3. If the root position is `InsufficientMaterial` (automatic): the score is
   `0`; choose a *stable legal* `best_move` fallback (e.g. the first
   legal root move) and leave `pv` empty.
4. If the root position is a claimable draw (`FiftyMoveClaim` /
   `ThreefoldClaim`): set a root score floor of `0`, but CONTINUE the depth
   loop so a winning move (if any) is found and reported with its real
   score and PV. If after the full search no line beats `0`, use the stable
   legal `best_move` fallback and an empty `pv`.
5. A root draw MUST have a `best_move` and MUST NOT panic on an empty PV.

#### 5.4.1 Root claim floor vs. `bestmove` — UCI / controller adjudication assumption

The current public interfaces cannot express "claim a draw" as an action:

* `SearchOutcome` is required to carry `best_move: Move` (a concrete move);
* the UCI thread must print `bestmove <token>` on every search result;
* there is **no** `ClaimDraw` action in `SearchOutcome`, and M3.1 does NOT
  add a UCI draw-*claim* output (see §1 non-goals).

Consequently, when the root's game-theoretic value is the claim floor `0` and
no real move achieves `>= 0`, the `best_move` is a **stable legal fallback**
(e.g. the first legal root move). This fallback is a **protocol placeholder
only**: M3.1 does NOT guarantee the fallback move itself implements a `0`
score, and it MUST NOT be described in code comments or tests as "correctly
choosing the claim". The engine assumes the **GUI / tournament controller**
reviews the real game history and adjudicates the threefold / fifty-move claim
when it applies the `bestmove`.

(If a future milestone adds `enum RootDecision { Move(Move), Claim(DrawReason)
}` and a UCI claim line, this assumption can be removed. Until then the
fallback is accepted, but it is explicitly NOT a claim action.)

### 5.5 `SearchPath` discipline

`classify_draw` / `is_threefold_repetition` receive `path.keys()` by `&`
(read-only) and never call `push_child` / `pop`. The M3.0 §16.8 push/pop
pairing is therefore untouched.

## 6. Repetition counting semantics (decision point — strict threefold)

**Search node:** a position is a claimable `ThreefoldClaim` only when its
Zobrist key occurs **at least THREE** times in the combined line
(`is_threefold_repetition` with `count >= 3`). The current Zobrist key is
the last element of `path.keys()`, so this counts the current node plus all
prior occurrences.

The earlier "second occurrence is a draw" convention is DELETED. FIDE
requires the *third* occurrence before a claim is valid; the second
occurrence is merely a position that *could* repeat, and either side may
still deviate. Relying on a second occurrence would:

* wrongly score KQ-vs-K as a draw on the second repeat, blocking a mate;
* create a false "dead position" that is not one;
* poison M3.2's TT if the draw context is cached incorrectly.

Locked tests:

```text
is_threefold_repetition(pos, &[key, key])        -> false
is_threefold_repetition(pos, &[key, key, key])   -> true
```

`is_threefold_in_game` uses the SAME `>= 3` helper over
`GameState::key_history()`.

## 7. Tests (acceptance gates)

### §N.1 `is_insufficient_material` (unit, legal FENs)

All FENs below parse (each has exactly one king per side) and are
side-independent — assert with both `w` and `b` to move where trivial.

* `7k/8/8/8/8/8/8/K7 w - - 0 1` → **true** (K vs K)
* `7k/8/8/8/8/8/8/KN6 w - - 0 1` → **true** (K+N vs K)
* `7k/8/8/8/8/8/8/KB6 w - - 0 1` → **true** (K+B vs K)
* `6bk/8/8/8/8/8/8/KB6 w - - 0 1` → **true** (K+B vs K+B, both bishops
  on dark squares — white B b1 parity odd, black B g8 parity odd)
* `5b1k/8/8/8/8/8/8/KB6 w - - 0 1` → **false** (bishops on opposite
  colors — white B b1 parity odd, black B f8 parity even — NOT automatic)
* `7k/8/8/8/8/8/8/KR6 w - - 0 1` → **false** (rook present)
* `7k/8/8/8/8/8/1P6/K7 w - - 0 1` → **false** (pawn present)
* `7k/8/8/8/8/8/8/KQ6 w - - 0 1` → **false** (queen present)
* `7k/8/8/8/8/8/8/KNN5 w - - 0 1` → **false** (K+NN vs K — can
  theoretically be mated, not a dead position)
* `7k/8/8/8/8/8/8/KBN5 w - - 0 1` → **false** (K+B+N vs K — not dead)
* `7k/8/8/8/8/8/8/KB1B1B3 w - - 0 1` → **true** (K + THREE same-color
  bishops vs K, from promotion — all on dark squares; proves arbitrary
  bishop count is covered)

### §N.2 `is_fifty_move_draw`

* `7k/8/8/8/8/8/8/K7 w - - 100 1` → **true**
* `7k/8/8/8/8/8/8/K7 w - - 99 1` → **false**

### §N.3 Threefold (strict >= 3)

* **(a) Predicate unit:** build a `Position` whose `zobrist_key()` is
  `key`; then
  - `is_threefold_repetition(&pos, &[key, key])` → **false**;
  - `is_threefold_repetition(&pos, &[key, key, key])` → **true**.
* **(b) Game-level (strict):** build a `GameState` by replaying the knight
  shuffle `g1f3 g8f6 f3g1 f6g8 g1f3 g8f6 f3g1 f6g8` from startpos. After
  these 8 half-moves the position before White's next move equals startpos,
  so the startpos key occurs at ply 0, after move 4, and after move 8 →
  exactly **3 times**. Assert
  `gs.key_history().iter().filter(|&&k| k == startpos_key).count() == 3`
  and `is_threefold_in_game(&gs)` is **true**. (The startpos key constant is
  `0x61A2_181F_8F2F_9B9C` per `zobrist.rs`.)
* **(c) Search-level (white-box, claim floor NOT forced 0):** construct
  `let mut path = SearchPath::new(vec![key, key, key]);` for a normal
  `pos` whose `zobrist_key() == key`, then call
  `negamax_impl(&mut pos, depth, 0, -INF, INF, &ctx, &limits, &mut pv,
  &mut path)`. Because the third occurrence is a *claimable* draw, expect
  `Some(s)` with `s >= 0` (the draw floor is respected, the search does
  not panic, and it is NOT forced to exactly `0` when a winning line
  exists). The two-occurrence case (`vec![key, key]`) must NOT be a draw:
  assert the search proceeds and returns a normal (possibly negative)
  `Some(...)`. (`negamax_impl` and `SearchPath::new` are reachable from the
  `search.rs` test module.)

### §N.4 Search integration returns correct scores

* **Automatic draw:** a K-vs-K position fed to `negamax_impl` returns
  `Some(0)`.
* **Terminal priority (P0-critical):** a checkmated position with
  `halfmove_clock == 100` fed to `negamax_impl` returns the **mate score**
  (negative, `Some(-(MATE - ply))`), NOT `Some(0)`. Use e.g.
  `k7/2K5/8/8/8/8/8/R7 b - - 100 50` (Black to move, in check from the
  a1-rook, no escape — checkmate; the fifty-move claim must NOT override
  the mate).
* **Claim is NOT a forced 0 (winning side):** `KQ vs K` with
  `halfmove_clock == 100` and the **winning** side to move —
  `7k/8/8/8/8/8/8/KQ6 w - - 100 50` — the search returns a score
  **`> 0`** (the engine still finds the mate; the claim floor does not
  replace a win).
* **Claim floor (losing side):** the same material with the **losing**
  side to move — `7k/8/8/8/8/8/8/KQ6 b - - 100 50` — the search
  returns a score **`>= 0`** (the side to move can claim, so the value is
  at least the draw floor; it is never a forced loss). The same floor
  semantics apply to a `ThreefoldClaim`.

### §N.5 Qsearch quiet-evasion reaches a claimable draw

`halfmove == 99`, the side to move is **in check**, and it has a **quiet**
legal evasion (no capture) that pushes `halfmove_clock` to `100`. E.g.
`4k3/8/8/8/8/8/4r3/4K3 w - - 99 50` (White Ke1, black rook e2 giving
check; Kd1/Kf1 are quiet evasions). The MOVER of this evasion owns the
intended-move claim (FIDE 9.3.1): before the evasion is "executed" it can
declare it and claim the fifty-move draw, so the evasion's edge score is
**exactly `0`** — not "the child side gets a `0` floor". Assert that
`quiescence_entered_impl` scores this evasion as `0` (via the move-edge
`claim_available_by_intended_move` check, §5.2), rather than silently
statically scoring it as a loss. This proves qsearch must check fifty-move /
threefold on the move edge, not only insufficient material at node entry.

### §N.6 Emergency-evasion child draw

A position in check with `halfmove_clock == 99` where at least one evasion
is a **quiet** move that pushes `halfmove_clock` to `100`. The MOVER of that
evasion owns the intended-move claim, so `search_final_evasion_ply` scores
that evasion's edge as `0` (via `claim_available_by_intended_move`, §5.3),
not via the static capture approximation, and the ply function returns a
valid `Option<i32>` without panic. This proves the per-child intended-claim
check in §5.3.

### §N.8 Root draw must not panic

Construct a root position that is a draw (e.g. K vs K, or KQ vs K with
`halfmove_clock == 100` and the losing side to move) and run
`search_best_move` / `search_best_move_with_history`. Assert:
* it returns `Some(out)` (no panic);
* `out.best_move` is a **legal** move (the stable fallback);
* `out.score` is `Some(0)` for insufficient material, or `Some(s)` with
  `s >= 0` for a claimable draw;
* `out.pv` MAY be empty;
* **do NOT assert that `out.best_move` itself has a `0`-score value** — per
  §5.4.1 the fallback is a protocol placeholder, not a claim action. The only
  hard guarantees are "no panic" + "legal move returned".

### §N.9 Fifty-move intended (prospective) claim

Construct a position with `halfmove_clock == 99`, side-to-move `X`, and a
single **quiet** legal move `m` that pushes `halfmove_clock` to `100`. Make
the child (after `m`) a position that, were it played *without* a claim, is
losing for `X` (e.g. `X` walks into a lost king-and-pawn endgame). Assert
the MOVE `m`'s parent-node score is **exactly `0`** — the mover claims the
fifty-move draw on the intended move and secures at least `0`, so this move
can never be scored as negative. Drive it through `negamax_impl` (white-box)
or `search_final_evasion_ply` / `quiescence_entered_impl` as appropriate,
and assert `Some(s)` with `s == 0` for that edge.

Example fixture (quiet move `Kd1` pushes `halfmove` 99→100; the resulting
child is otherwise lost for White):
`4k3/8/8/8/8/8/3r4/3K4 w - - 99 50` — White to move at `halfmove 99`,
the only quiet evasion is `Kd1` (rook on d2 gives check; `Kd1` steps but
keeps `halfmove` 99→100). Assert the `Kd1` edge scores `0` for White even
though the child is materially lost.

### §N.10 Threefold intended (prospective) claim

Build a `path` whose key slice contains the current position's key exactly
**twice** (so the current node is NOT yet a draw), plus a child key that,
after `make_move(m)` + `push_child`, becomes the **third** occurrence. Assert
that the move `m`'s edge scores **exactly `0`** (`claim_available_by_intended_move`
true), even though the resulting child, played without a claim, might be
disadvantageous. This proves the prospective claim is detected on the edge,
not deferred to the child's own claim floor.

### §N.11 Immediate mate precedence over intended claim

A move that simultaneously (a) satisfies a fifty-move / threefold claim
condition AND (b) delivers checkmate must return the **mate score**, never
`0`. Construct: `halfmove_clock == 99` (or a two-occurrence path) with a
move `m` that makes `halfmove` 100 (or the third repetition) AND is
checkmate. Assert the edge scores the mate value (positive from the mover's
perspective: `MATE - (ply+1)`), confirming `child_moves.is_empty()`
(terminal) is checked **before** `claim_available_by_intended_move`. This
guards the ordering in §5.1 / §5.2 / §5.3.
