## 1. Core design: keep three SearchProfile generations

The existing two profiles are insufficient for a trustworthy M4.2 A/B,
because:

```
M4Reference = M4.0
Current     = M4.1
```

Once `Current` enables PVS, the directly-replayable M4.1 baseline is lost.

Therefore add:

```rust
// src/engine/search.rs (pub(crate))
pub(crate) enum SearchProfile {
    M4Reference,
    M41Reference,
    Current,
}
```

Precise definitions:

```
M4Reference
    M4.0 full-window alpha-beta
    no killer / history
    preserves the 1149 / 963 / 498,474 historical baseline

M41Reference
    M4.1 full-window alpha-beta
    uses killer / history
    no PVS
    preserves the 236,418-node M4.1 A/B baseline

Current
    M4.1 killer / history
    + M4.2 PVS
    UCI production path continues to use this profile
```

Implementation rules:

- `SearchHeuristics` is created by `M41Reference` and `Current`.
- The M4.1 seven-level ordering is shared by `M41Reference` and `Current`.
- PVS is enabled ONLY in `Current`.
- `M4Reference` MUST NOT carry any M4.1 / M4.2 per-node overhead.
- The original entry `search_best_move_with_history_and_tt()` keeps using
  `M4Reference` exclusively.
- UCI still calls the profile-aware entry with `Current`.
- No new UCI option.

## 2. PVS window contract

For the parent window:

```
[alpha, beta]
```

### 2.1 First move

The first ordered move MUST use the full window:

```
child window = [-beta, -alpha]
```

### 2.2 Later moves

Later moves first use a null window:

```
parent scout window = [alpha, alpha + 1]
child scout window  = [-(alpha + 1), -alpha]
```

A scout is used ONLY when ALL of the following hold:

```
profile == Current
not the first move at this node
alpha + 1 < beta
child is not a probe-confirmed terminal / intended-claim exact result
```

A safe addition MUST be used:

```rust
let Some(scout_beta) = alpha.checked_add(1) else {
    // cannot form a safe narrow window; fall back to the full window
};
```

If:

```
scout_beta >= beta
```

then the caller itself is already a null-window node; search it once with
the caller's original window and do NOT manufacture a "narrower window".

Never use a raw `alpha + 1` that could overflow.

## 3. Re-search conditions

After scouting a later move, the parent-view score is `score`.

### No re-search: fail-low

```
score <= alpha
```

The move cannot improve the current best value.

### Full re-search: improves alpha but does not cutoff

```
alpha < score < beta
```

Use the original effective window saved before this move was searched:

```
child full window = [-beta, -alpha_before_move]
```

The full re-search result is the move's final score and PV source.

### No re-search: fail-high

```
score >= beta
```

The scout already proves the beta cutoff holds; no full re-search needed.

A legal lower-bound score returned by the scout may be used to complete
the cutoff.

### Update timing

The following operations MUST each be performed exactly once, based only
on the move's FINAL search result:

- `best` update
- `alpha` update
- `node_best_move` update
- `pv.set_from_child()`
- killer update
- history update
- beta-cutoff decision

Must NOT:

- update once after the scout improves alpha, then update a second time
  after the re-search;
- reward history twice for the same quiet cutoff;
- commit the soon-to-be-fully-researched scout PV into the parent PV.

## 4. Non-root negamax integration

PVS is first applied to:

```
depth > 0
non-root ordinary negamax
profile == Current
```

Keep the current seven-level ordering unchanged:

```
TT move
promotions
captures / en passant
killer 0
killer 1
history quiets
deterministic tie-break
```

The first move is produced by the order above and searched with the full
window; remaining moves follow the PVS rule.

### 4.1 ChildProbe

The first search of a move continues to use the current:

```
probe_child_draw()
```

It is responsible for:

- obtaining a precise node budget for a node;
- terminal child;
- intended fifty / threefold claim;
- child PV row cleanup.

If the first probe returns:

```
Terminal
IntendedClaim
```

the result is already exact; re-search is NOT allowed.

If it returns:

```
Continue
```

and the scout triggers a full re-search, then, while keeping the child
position and SearchPath already pushed:

1. obtain a real re-search node again via `try_enter_node()`;
2. call `negamax_entered_impl()`;
3. use the full window;
4. the researched node entry cleans up its own PV row;
5. on re-search abort, first restore path and position, then propagate
   `None` upward.

Every scout and every re-search is a real node visit and MUST each be
counted against the node budget.

### 4.2 Abort

An abort may occur at:

- before the scout starts;
- inside the scout;
- after the scout completes, before the re-search obtains a node;
- inside the re-search.

Each case MUST guarantee:

```
move is unmade
SearchPath is popped
Position / FEN / Zobrist fully restored
current partial iteration does not enter the final result
no partial parent TT entry is written
root keeps the previous completed iteration
```

Legal TT bounds written by an already-completed scout search may be kept;
what is forbidden is writing a pseudo-complete result for the not-yet-finished
parent node.

## 5. Root PVS

Root PVS is implemented independently of the non-root path, as a separate
commit.

Current root:

- uses the TT hash move and the previous-iteration best move;
- has no root beta cutoff;
- iterates over all root moves.

M4.2 keeps these behaviors.

### 5.1 Root rule

First root move:

```
full window
```

Later non-terminal root moves:

```
scout first
full re-search when score > alpha
```

Because the root beta is a safe +infinity sentinel and any normally
reachable score is below it, the root condition is equivalent to:

```
score > alpha → full re-search
```

but the code SHOULD still use the general condition:

```
score > alpha && score < beta
```

Terminal and intended-claim root moves are exact results; no re-search.

### 5.2 Root result exactness

After PVS, it is no longer required that "every failed root move gets an
exact score". What MUST be guaranteed:

- all root moves are still visited;
- fail-low root moves may keep only their upper-bound nature;
- any root move that could become the new best is fully re-searched;
- the final `RootIteration.score` is an exact score;
- the final bestmove and PV come from a full search or an exact terminal
  result;
- the root TT entry is still stored as Exact after the full iteration
  completes;
- the claimable-root 0-score floor is unchanged;
- no root beta cutoff is added.

The original "real scores for every root move" comment should be corrected
to:

```
all root moves are checked;
moves that may improve alpha are fully re-searched;
the final root best score is exact.
```

## 6. PV contract

The most common PVS correctness bug is not the score but residual PV.
Locked rules:

- scout fail-low and does not improve alpha: MUST NOT copy that child PV;
- scout fail-high causes cutoff: the cutoff move and its legal scout line
  may be recorded;
- scout triggers full re-search: finally only the child PV produced by
  the full re-search may be copied;
- the full re-search entry MUST clear the original scout child row;
- re-search abort: MUST NOT disguise the scout PV as a full PV;
- completed root PV MUST be stepwise-legal;
- a fallback move MUST NOT be disguised as a real PV;
- an empty-PV draw-claim placeholder remains allowed.

## 7. TT contract

MUST NOT modify:

```
TtKey
Exact / Lower / Upper meaning
mate-ply codec
depth-preferred replacement
hash-move legality check
TT capacity and UCI lifecycle
```

Current TT classifies bound by the caller's window; at equal depth an Exact
may replace a non-Exact. This rule is sufficient to support PVS's TT
contract: bound is classified relative to the caller window saved when the
node was entered, and the parent stores at most once — only after the node
completes normally or cuts off normally.

Locked:

- a scout search stores Upper / Lower / Exact by its scout call window;
- a full re-search reclassifies by its call window;
- the parent stores once, only after normal completion or normal cutoff;
- the parent bound is still classified relative to the caller window saved
  when the node was entered;
- when a draw floor raises the effective alpha, TT classification still
  uses the caller window from before the draw floor;
- an abort MUST NOT write a TT entry for the unfinished node;
- do NOT change the TT replacement strategy for PVS.

If implementation reveals a NEED to modify `tt.rs`, STOP immediately and
re-review the scope; do NOT modify it casually.

## 8. Draw, mate, and special results

MUST keep:

- terminal before draw;
- insufficient material auto 0;
- fifty-move and threefold are optional 0-score floors;
- intended-move claim exact 0;
- mate distance and sign unchanged;
- claimable root with no winning move returns 0-score fallback and empty PV;
- a claimable root where scout yields `score <= 0` needs no re-search;
- a claimable root where scout yields `score > 0` MUST be fully re-searched;
- qsearch implements NO PVS at all.

"qsearch implements no PVS" does NOT mean it always receives a full window:
a parent negamax scout may enter the depth-0 qsearch with a narrow window,
but qsearch's own algorithm, ordering, and recursion stay unchanged.
