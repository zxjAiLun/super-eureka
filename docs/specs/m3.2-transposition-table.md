## 1. P0: plain Zobrist key is insufficient as TT identity

It must be explicitly forbidden to use:

```rust
TT key = pos.zobrist_key()
```

as the sole identity.

The same board Zobrist key can correspond to different search values:

1. halfmove `0` vs `100`: the former cannot claim; the latter has a 0 floor.
2. halfmove `98` vs `99`: a quiet child of the latter can form an intended fifty-move claim.
3. Current key appears once vs three times: the latter has a current threefold floor.
4. Even if the current key count is the same, occurrence counts of *other* keys in the history differ: a given child may be the third occurrence in one history but not another.

Therefore it is forbidden to:

- Pack only the current repetition count into an entry.
- Disable TT only for draw nodes.
- Disable TT only for halfmove >= 99.
- Ignore the prospective threefold context.

None of these are sufficient for ordinary-node value reuse.

## 2. Context-safe TT key

Define the logical identity:

```rust
struct TtKey {
    position: ZobristKey,
    halfmove_clock: u32,
    repetition_signature: u64,
}
```

- `position`: the existing board Zobrist (pieces, side to move, castling rights, effective EP target).
- `halfmove_clock`: the exact halfmove counter (NOT a boolean `>=100`).
- `repetition_signature`: a commutative multiset signature over the entire `SearchPath` — captures every Zobrist key's occurrence count regardless of order.

**Not included:**
- fullmove number.
- search depth.
- ply.
- alpha/beta.
- node counter.

Every TT entry must store the full `TtKey` for hit verification. It is insufficient to store only a table index or a short verification tag.

## 3. Repetition signature

`SearchPath` must eventually grow to include:

```rust
struct SearchPath {
    history: Vec<ZobristKey>,
    counts: HashMap<ZobristKey, usize>,
    repetition_signature: u64,
}
```

or an equivalent structure.

The signature is the XOR of tokens for every key-count pair where count > 0:

```text
signature = XOR(token(key, count))
            for every key with count > 0
```

XOR is its own inverse, so the update is a simple identity:

```text
signature ^= token(key, old_count)   // remove old
signature ^= token(key, new_count)   // add new
```

A count of `0` never contributes a token (`xor 0` is a no-op, so the remove step for `1 -> 0` is included for clarity and the add step is simply skipped).

### Token function

Fixed, deterministic 64-bit mixing with domain separation per SplitMix64:

```rust
fn token(key: ZobristKey, count: usize) -> u64 {
    let count64 = u64::try_from(count)
        .expect("usize always fits u64 on supported Rust targets (32/64-bit)");
    let mut z = key as u64 ^ count64.wrapping_mul(0x9e3779b97f4a7c15);
    z ^= z >> 30;
    z = z.wrapping_mul(0xbf58476d1ce4e5b9);
    z ^= z >> 27;
    z = z.wrapping_mul(0x94d049bb133111eb);
    z ^= z >> 31;
    z
}
```

The `count` is domain-separated by the multiplicative constant.

All arithmetic is `u64 wrapping` (mod 2^64). No `+=`/`-=` on the signature itself — only `^=` (XOR). Wrapping is naturally correct for XOR.

Occurrence count uses `usize` because it is bounded by `history.len()`, which is itself `usize`. There is no independent `u32` overflow risk.

### Incremental operations

```text
push_child(key):
    old = counts.get(key).copied().unwrap_or(0)   // usize

    if old > 0:
        signature ^= token(key, old)

    new = old + 1                                  // usize, naturally unbounded
    counts.insert(key, new)
    signature ^= token(key, new)

    history.push(key)

pop():
    key = *history.last().unwrap()

    new = counts[key]                              // usize
    signature ^= token(key, new)

    if new == 1:
        counts.remove(key)
    else:
        old = new - 1
        counts.insert(key, old)
        signature ^= token(key, old)

    history.pop()

restore_root(root_len):
    assert 1 <= root_len <= history.len()
    while history.len() > root_len:
        pop()

keys():
    &history

last():
    history.last()

repetition_signature():
    self.repetition_signature

occurrences(key):
    self.counts.get(key).copied().unwrap_or(0)
```

`restore_root` MUST NOT use `Vec::truncate` alone, because that would leave `counts` and `repetition_signature` out of sync. Each iteration of the while-loop calls the full `pop()` which restores all three fields atomically.

### Invariants

```text
history.last() == current Position key
counts exactly match a fresh count of every key in history
signature exactly matches a fresh XOR rebuild from counts
push then pop restores all three to pre-push state
restore_root(n) restores all three to the state at length n
clone preserves all three
```

### Collision model

The deterministic acceptance corpus must distinguish different occurrence counts. As with the existing Zobrist key, theoretical 64-bit collisions remain possible; the spec does not claim mathematical injectivity.

### Forbidden

- Random seed.
- Unstable `DefaultHasher` implementation details.
- History-order hash.
- Re-scanning the entire path per node in the final implementation.
- Ordinary `u64 +=` / `u64 -=` on the signature (use XOR only).
- Relying on debug-vs-release overflow behaviour differences.

## 4. TT entry and Bound

```rust
enum Bound {
    Exact,
    Lower,
    Upper,
}

struct TTEntry {
    key: TtKey,
    depth: u32,
    score: i32,          // TT-encoded mate score
    bound: Bound,
    best_move: Option<Move>,
}
```

First version explicitly excludes:

- Evaluation-only entries.
- Static eval storage.
- Generation aging.
- Node-type / PV-node flags.
- Quiescence depth.
- Singular extension data.

`best_move == None` is a legal state, used for:

- A current-claim floor that itself was stored as the node value.
- A root claim placeholder.
- A node with no real move achieving the stored score.

The root protocol fallback must NOT be stored as a hash move.

## 5. Table structure and memory

First version: single-slot direct-mapped table:

```rust
Vec<Option<TTEntry>>
```

or an equivalent no-unsafe fixed-slot structure.

Public API:

```rust
TranspositionTable::disabled()
TranspositionTable::new_mb(mb: usize) -> Result<Self, TtAllocError>
resize_mb(&mut self, mb: usize) -> Result<(), TtAllocError>
clear(&mut self)
probe(&self, key: TtKey) -> Option<TTEntry>
store(&mut self, entry: TTEntry)
capacity_entries(&self) -> usize
size_mb(&self) -> usize
```

Requirements:

- `disabled()` has capacity 0; `probe` always misses; `store` is a no-op.
- All byte/entry arithmetic is checked.
- Allocation failure must NOT panic.
- Resize failure preserves the old table and its configuration.
- Successful resize always empties the table.
- At least 1 entry must be allocated.
- No `unsafe`.
- Index must mix `position`, `halfmove_clock`, and `repetition_signature` — NOT only the low bits of `position`.

### Allocation contract (checked arithmetic)

```text
bytes = mb.checked_mul(1024 * 1024)
entries = bytes / size_of::<Option<TTEntry>>()
entries >= 1
```

Allocation uses `try_reserve_exact` on a temporary `Vec`; slots are filled (`None`) only after reserve succeeds. The old table is replaced only after the new allocation is ready.

- Allocation failure (`try_reserve_exact` returns Err): old table, old `size_mb`, old entries all preserved.
- Resize success: new table is empty.
- `size_mb()` returns the configured MB (not the byte-rounded-down value inferred from entry count).
- `disabled()` is the ONLY legal 0-MB / 0-entry state.

### TtAllocError

```rust
enum TtAllocError {
    InvalidSize,
    ArithmeticOverflow,
    AllocationFailed,
}
```

Locked zero-size semantics:

```text
TranspositionTable::new_mb(0)      -> Err(InvalidSize)
enabled_tt.resize_mb(0)            -> Err(InvalidSize), old table preserved
TranspositionTable::disabled()     -> only valid zero-capacity constructor
Default::default()                 -> disabled()
```

UCI layer always clamps `setoption name Hash value 0` to 1 MB before calling `resize_mb`, so `InvalidSize` is never triggered by normal UCI usage.

## 6. Depth-preferred replacement

### Empty slot
Write directly.

### Same full key
Replace when ANY of the following holds:

- New depth > old depth.
- Same depth, new bound is `Exact`, old bound is not.
- Same depth and bound quality, but the new entry has a real `best_move` while the old entry does not (`best_move` is `None`).

A shallower non-exact entry must NOT overwrite a deeper exact entry.

### Different full key, same index (collision)
Replace only when:

```text
new depth >= old depth
```

No generation aging in M3.2 first version; `ucinewgame` clears the table.

## 7. Mate score ply codec

TT mate scores must be independent of the current root ply.

### Constants

```rust
const MATE: i32 = 1_000_000;
const MATE_THRESHOLD: i32 = MATE - 1000;  // 999_000
const MAX_MATE_PLY: u32 = 999;
```

### Recognition

```text
score >  MATE_THRESHOLD  -> positive mate
score < -MATE_THRESHOLD  -> negative mate
otherwise                -> ordinary cp score (unchanged)
```

Exactly `±MATE_THRESHOLD` is an ordinary cp score, not a mate.

### Codec

Returns `Option<i32>`.

Both `score_to_tt` and `score_from_tt` must reject `ply > MAX_MATE_PLY` before any score arithmetic:

```rust
fn score_to_tt(score: i32, ply: u32) -> Option<i32> {
    if ply > MAX_MATE_PLY { return None; }

    if score > MATE_THRESHOLD {
        score.checked_add_unsigned(ply)
    } else if score < -MATE_THRESHOLD {
        score.checked_sub_unsigned(ply)
    } else {
        Some(score)
    }
}

fn score_from_tt(stored: i32, ply: u32) -> Option<i32> {
    if ply > MAX_MATE_PLY { return None; }

    if stored > MATE_THRESHOLD {
        stored.checked_sub_unsigned(ply)
    } else if stored < -MATE_THRESHOLD {
        stored.checked_add_unsigned(ply)
    } else {
        Some(stored)
    }
}
```

These methods (`checked_add_unsigned`, `checked_sub_unsigned`) exist on `i32` since Rust 1.66. Forbidden:

```text
wrapping_add_unsigned
wrapping_sub_unsigned
```

`wrapping_*` never returns `None`, making it impossible to honour the `Option` contract. `checked_*` guarantees:

- Return `None` on i32 overflow → store caller skips the entry (does not write to TT).
- Return `None` on decode overflow → probe treats it as a miss (TT entry is considered invalid for this context).
- No panic on overflow for any input.
- Ply check enforced inside the codec, not left to the integration layer alone.

### Covered cases

| Input | Expected |
|---|---|
| `score_to_tt(MATE_THRESHOLD, 0)` | `Some(999_000)` (ordinary cp) |
| `score_to_tt(MATE_THRESHOLD + 1, MAX_MATE_PLY)` | `Some(999_001 + 999 = 1_000_000)` (mate) |
| `score_to_tt(MATE_THRESHOLD + 1, MAX_MATE_PLY + 1)` | `None` (ply > limit) |
| `score_to_tt(i32::MAX, 1)` | `None` (overflow) |
| `score_to_tt(MATE_THRESHOLD + 999_999, 1)` | `None` (overflow, exceeds i32::MAX) |
| `score_to_tt(-MATE_THRESHOLD - 1, MAX_MATE_PLY)` | `Some(-999_001 - 999 = -1_000_000)` (negative mate) |
| `score_to_tt(-(MATE_THRESHOLD + 1), MAX_MATE_PLY + 1)` | `None` (ply > limit) |
| `score_to_tt(i32::MIN, 1)` | `None` (overflow) |
| `score_from_tt(valid_stored, MAX_MATE_PLY + 1)` | `None` (ply > limit) |
| Positive mate same-ply roundtrip | restored to `MATE_THRESHOLD + remainder` |
| Negative mate same-ply roundtrip | restored to `-(MATE_THRESHOLD + remainder)` |
| Different-ply decode | correct remaining distance |
| Pure cp score unchanged through roundtrip (both ends) | `Some(score)` |
| `ply == MAX_MATE_PLY` | succeeds (boundary) |
| `ply == MAX_MATE_PLY + 1` | `None` (out of range) |

### Threshold boundary

```text
+999000  = ordinary cp score
+999001  = mate (in-1)
...
+999999  = mate (in-999)
-999000  = ordinary cp score
-999001  = mate (in-1)
...
-999999  = mate (in-999)
```

`MATE_THRESHOLD` (±999_000) unchanged — still ordinary cp. One point beyond (±999_001) is mate.

## 8. Search ordering (negamax_entered_impl)

Locked order:

```text
Node already counted by try_enter_node
pv.clear_at(ply)

generate legal moves
terminal mate/stalemate FIRST

save caller_alpha / caller_beta
apply insufficient / fifty / threefold semantics

construct context-safe TtKey
probe TT

if NOT cutoff:
    depth == 0 -> existing qsearch (no TT probe/store)
    order hash move first (if legal)
    search move loop
    after loop: store
```

Rationale:

- TT must NOT mask terminal states.
- Draw-floor semantics must apply **before** the TT window decision.
- A TT hit still consumes one real node.
- Before TT cutoff returns, the child PV row must already be cleared by `pv.clear_at(ply)` (the parent is responsible for its own PV row; the child row may have unrelated stale data).
- An aborted node (propagated `None`) must NOT write to TT.

## 9. Probe semantics

If the full `TtKey` is not equal → miss.

If `entry.depth < requested_depth`:

- Score cutoff is NOT allowed.
- `best_move` may still be returned for ordering.

If depth is sufficient:

```text
Exact:
    return decoded score

Lower:
    cutoff ONLY when score >= beta

Upper:
    cutoff ONLY when score <= effective alpha
```

Preserve the current negamax fail-soft semantics. Do NOT unconditionally convert every cutoff to `beta`/`alpha`.

No alpha/beta window tightening in the first version. Only the direct cutoffs above, to minimise semantic surface.

`caller_alpha` must be saved before the draw-floor modifies `alpha`. The final Bound classifier uses the original caller window:

```text
score <= caller_alpha -> Upper
score >= caller_beta  -> Lower
otherwise             -> Exact
```

This prevents a current-claim floor that raises `best` to `0` from being mis-stored as `Upper`.

## 10. PV and TT cutoff

The TT stores exactly one move, never a full PV.

Locked:

- On TT score cutoff, the current PV row remains empty.
- If the parent node decides to pick this child as its best, the PV may legitimately truncate to `[parent_move]`.
- A stale sibling tail must NOT be inherited.
- The first version does NOT reconstruct a full PV by following TT chain links.
- A UCI PV may be shorter than the search depth, but every move in it must be legal.
- `bestmove` must NOT be derived from an empty PV.

## 11. Hash move ordering and tie-breaking

The hash move may be used for ordering ONLY if it actually appears in the current `generate_legal_moves` result. It is forbidden to execute an unverified TT move.

Order:

```text
legal hash move first
remaining moves: existing MVV-LVA / promotion stable sort
```

The hash move must NOT:

- Lose any other legal moves.
- Appear twice.
- Change the root claim fallback.
- Have the protocol placeholder stored as a real best move.

### Tie-breaking semantics

Moving the hash move to the front changes which move is searched first. The current search uses `if score > best_score` (strictly greater), so when two moves have the **same** score, the one searched first wins. Hash-move ordering therefore changes tie-breaking.

M3.2 first version does NOT introduce canonical tie-breaking. The acceptance model is:

```text
Required (TT enabled vs disabled):
    same minimax score
    returned bestmove is legal
    PV is legal
    terminal/draw/mate semantics match

Allowed to differ:
    bestmove and PV may differ when multiple root moves share the same score,
    because hash-move ordering changes the first-searched tie resolution
```

Cold search (freshly allocated or cleared table before the call) on the §16.9 fixtures:
- Same score, legal bestmove, legal PV.
- The specific bestmove `b1c3` / `e4a4` is a **deterministic implementation-regression expectation** for a cleared table on this engine version, NOT a proof of chess uniqueness.

Warm or persistent search on any fixture:
- Same minimax score, legal bestmove, legal PV, same terminal/draw/mate semantics.
- bestmove/PV may differ when root moves tie — no mechanical requirement for `b1c3` or `e4a4`.

The spec shall NOT describe any fixture as "unique best move" or "single-best-move" without a white-box root-score uniqueness test (see §18).

Root stable fallback remains independent of hash ordering (captured before any re-ordering).

### Root layer, first version

- A TT Exact hit alone may NOT skip the entire iterative-deepening root search.
- A legal hash move may be used for ordering.
- A completed root iteration may store an Exact entry.
- A claim placeholder entry must have `best_move = None`.

## 12. Qsearch boundary

First version does **NOT** directly probe or store quiescence nodes.

Locked:

- The final qsearch result at `depth == 0` in regular negamax may be stored as a depth-0 regular TT entry.
- `quiescence_entered_impl` itself does NOT read or write TT.
- Regular depth and qply are NOT mixed.
- `search_final_evasion_ply` does NOT read or write TT.

Any future qsearch TT must be specified separately.

## 14. UCI TT ownership

Single-search-thread architecture:

```rust
Arc<Mutex<TranspositionTable>>
```

The UCI main loop holds one persistent table.

`spawn_search` clones the `Arc`; the search thread acquires the mutex when it begins and holds the sole mutable TT borrow for the entire search run.

Forbidden:

- Lock/unlock per node.
- Create a new table per `go`.
- Multiple active searches writing to TT concurrently.

Existing logic guarantees the engine stops and joins before any command modifies state.

If the mutex is poisoned:

```text
match tt.lock():
    Ok(guard) -> use guard normally
    Err(poisoned):
        guard = poisoned.into_inner()
        guard.clear()
        print info string "info string TT mutex poisoned; recovered"
        continue with same configured size
```

Do NOT attempt to re-allocate in poison recovery (avoiding a second failure path). The engine continues working.

## 15. UCI Hash option

`uci` outputs before `uciok`:

```text
option name Hash type spin default 16 min 1 max 1024
```

Only the following `setoption` is handled:

```text
setoption name Hash value N
```

Parsing requirements:

- Option name is case-insensitive.
- Correctly handle `name` / `value` tokens.
- Missing, non-numeric, zero, or out-of-range values must NOT panic.

Clamp semantics (not rejection):

```text
numeric N:
    clamp N to 1..=1024

missing value:
    table unchanged
    output diagnostic: info string invalid Hash value

non-numeric value:
    table unchanged
    output diagnostic: info string invalid Hash value

parsed as zero:
    clamped to 1 MB

parsed above 1024:
    clamped to 1024 MB
```

Allocation failure preserves the old table and outputs:

```text
info string unable to resize Hash table
```

Successful resize leaves the table empty.

When `setoption` arrives:

```text
stop_and_join active search
resize + clear TT
GameState unchanged
```

Unknown options are silently ignored; no panic.

## 16. Lifecycle

### `go`
Reuses the current persistent TT.

### `position ...`
Does NOT clear TT. The context-safe key guarantees that different history / halfmove will not produce false hits.

### `ucinewgame`

```text
stop_and_join
GameState::startpos()
TT.clear()
```

The configured Hash MB is preserved.

### `setoption Hash`

```text
stop_and_join
resize + clear
```

### `quit`
Stop/join then normal drop.

### `isready`
Responds `readyok` immediately; must NOT wait for or lock the TT.

## 18. Acceptance tests

### SearchPath / repetition signature

- Constructor from history with duplicate keys builds correct counts and signature.
- Single `push_child` and `pop` restores all three fields.
- Nested push/pop (depth 3) restores all three fields.
- `restore_root` from a deep child fully restores history, counts, and signature.
- Abort-style restore (push 4, pop 2 via restore_root, then continue) works.
- Fresh rebuild signature (re-scan) equals incremental signature.
- Same current-key count but a child-key count differs: signature differs from a reference fixture.

### Core table

- Disabled probe/store: always miss, never store.
- Exact key hit: full `TtKey` matches.
- Context-key mismatch: different `TtKey` → miss (halfmove, repetition signature).
- Index collision: full-key verification rejects mismatched entry.
- `clear` empties.
- `resize_mb` succeeds after clear; failure preserves old table.
- Allocation arithmetic boundaries.
- Same-key replacement (depth-preferred, Exact-over-Lower, `best_move`-over-`None`).
- Depth-preferred collision (different keys, same slot).
- Mate score roundtrip (positive, negative, at different plies, cp unchanged).
- `new_mb(0)` returns `Err(InvalidSize)`.
- `resize_mb(0)` on an enabled table: returns `Err(InvalidSize)`, old table size and entries preserved.
- `Default::default()` is equivalent to `disabled()` (capacity 0, probe always miss, store no-op).

### Mate codec boundaries

- `MATE_THRESHOLD` unchanged through codec.
- `MATE_THRESHOLD + 1` encoded and decoded as mate.
- `-MATE_THRESHOLD` unchanged through codec.
- `-MATE_THRESHOLD - 1` encoded and decoded as mate.
- Positive mate same-ply roundtrip.
- Negative mate same-ply roundtrip.
- Different-ply mate-distance shift.
- `MAX_MATE_PLY` boundary.
- Overflow input (`ply > MAX_MATE_PLY` or stored leading to overflow) → `None`, no panic.
- `score_to_tt(MATE_THRESHOLD + 1, MAX_MATE_PLY)` succeeds (boundary case).
- `score_to_tt(MATE_THRESHOLD + 1, MAX_MATE_PLY + 1)` == `None`.
- `score_from_tt(valid_stored, MAX_MATE_PLY + 1)` == `None`.
- `score_to_tt(i32::MAX, 1)` == `None` (positive mate overflow).
- `score_to_tt(i32::MIN, 1)` == `None` (negative mate overflow).
- Forbidden: `wrapping_add_unsigned` / `wrapping_sub_unsigned` in codec.

### Draw-context isolation

Same board Zobrist key, different `TtKey` → must MISS:

- halfmove 0 vs 99.
- halfmove 99 vs 100.
- Repetition signature: onefold vs threefold of current key.
- Repetition signature: current key count same but a child-key count differs.

### Root-score uniqueness test (optional, for any fixture claimed unique)

```text
search every legal root move independently at the locked depth
collect exact minimax scores
assert exactly one move has the maximum score
```

Until this test is written for a fixture, the spec shall NOT describe it as "unique" or "single-best-move".

### Search correctness (TT enabled vs TT disabled)

```text
Required:
    same minimax score
    returned bestmove is legal
    PV is legal (every move in PV is a legal move at that position)
    terminal / draw / mate semantics identical
    mate distance identical
    Position / FEN / Zobrist / SearchPath restored identically

Allowed to differ when multiple moves share the same score:
    bestmove and PV may differ (hash-move ordering changes tie-breaking)
```

### Equal-score fixture

A position where two or more root moves have identical score:

```text
Required:
    TT enabled and disabled return the same score
    both bestmoves are legal
    if bestmoves differ, the test accepts both as correct
    the stable fallback is independent of hash ordering
```

### TT effectiveness

- Second (warm) search on a deterministic fixture: `nodes` <= first search.
- At least one transposition-rich fixture with strictly reduced nodes.
- No universal node monotonicity guarantee is made for arbitrary positions, because direct-mapped collisions and hash ordering can alter the searched tree.

### Bound (white-box)

- Narrow window: `Exact` storage and retrieval.
- `Lower` cutoff when `score >= beta`.
- `Upper` cutoff when `score <= effective alpha`.
- Insufficient depth: only hash move returned, no score cutoff.
- Decoded mate bound preserves correct distance.

### Abort

- Aborted parent does NOT write a partial entry.
- Completed child entry may survive.
- Stop/node/deadline restoration unchanged.

### Root claim

- Claim placeholder (Exact 0) must NOT store a fallback hash move.
- Real positive-score move may store a hash move.
- Empty PV does not panic.

### UCI

- `uci` declares `Hash` option.
- Valid `setoption` changes size.
- Malformed `setoption` does not panic.
- `setoption Hash 0` clamped to 1 MB.
- `setoption Hash 2048` clamped to 1024 MB.
- Allocation failure leaves size unchanged.
- Resize clears the table.
- `ucinewgame` clears but preserves size.
- `position` does NOT clear.
- `setoption` during active search stops and joins first.
- `isready` responds immediately.
