# M4.0 — Search Measurement Baseline

**Status:** CLOSED
**Initial base:** `159f718`
**Closed implementation commit:** `5b19311d066e9c74d8604a83b4ec6157928815bf`

Milestone 3 (draw rules + context-safe transposition table) is closed. M4 is the
*measurement* milestone: before any search optimization (killer/history, PVS,
aspiration, move-list allocation, incremental eval, bitboard, …) we need a
trustworthy, repeatable measuring system so every later change can be compared
against a fixed reference.

M4.0 deliberately does **not** improve the search. It only adds an
independent benchmark harness that drives the *existing* search entry points and
records results. No search control flow, scoring, ordering, TT behavior, or
public search API changes.

## 1. Goal

Establish a search measurement baseline. The harness must clearly separate:

```text
fixed-depth node count      → measures search-tree efficiency
fixed-nodes elapsed/NPS  → measures per-node throughput
TT disabled / cold / warm → separates baseline, first search, persistent-TT gain
correctness fields         → score, bestmove, PV, full restoration
machine-dependent fields    → elapsed, NPS
```

After M4.0, every later optimization (killer/history, PVS, aspiration,
move-list allocation, incremental eval, bitboard) must be comparable using the
same command set.

## 2. Core principles

### 2.1 No change to strength or search semantics

Must NOT change in M4.0:

- move generation
- evaluation
- move ordering
- alpha-beta control flow
- quiescence
- TT probe / store / replacement
- mate codec
- draw rules
- node accounting
- UCI thread and `Hash` lifecycle
- existing public search API behavior
- existing bestmove / score / PV / exact node baselines

A new, independent benchmark module that *collects results from* the existing
search entry is allowed.

### 2.2 Default tests must stay fast

`cargo test` / `cargo test --release` must NOT contain full KQK/KRK
self-play, 50-move playouts, or long self-play. M4.0 runs **single-position
searches** only — never dozens of consecutive games.

Rules:

- benchmarks run via an explicit CLI command;
- ordinary unit tests only cover argument parsing, formatting, fixture validity,
  and very shallow searches;
- no slow tests in the default gate;
- no wall-clock assertions in unit tests;
- no commit gate may become a multi-minute affair.

### 2.3 Measurement data must be transparent

No opaque checksum or single unexplained total. Every result retains:

```text
fixture
mode
limit
repeat
score
bestmove
completed_depth
stopped
nodes
elapsed_us
nps
PV
```

Correctness data and performance data are explained separately.

## 4. Implementation structure

New module:

```text
src/engine/bench.rs
```

Responsibilities:

- fixture definition;
- benchmark argument parsing;
- disabled / cold / warm modes;
- `Position` and history construction;
- calling the existing history-aware, TT-aware search entry;
- result validation;
- timing;
- per-result and summary output.

`src/main.rs` gets a new `bench` subcommand branch near the existing
`perft` branch. All other launch paths keep entering the UCI loop.

Because the binary must cross the lib boundary, a strictly limited entry is
exposed:

```rust
pub fn run(args: &[String]) -> Result<(), String>
```

The TT-aware search entry itself is **not** widened into a new public API.
The harness calls the already-`pub(crate)` entry
`search::search_best_move_with_history_and_tt(pos, history, limits, ctx, &mut tt)`.

`engine/mod.rs` only adds the module declaration; it must NOT re-export a
new set of search internals.

The existing entry points used (unchanged):

```rust
// crate::engine::search
pub(crate) fn search_best_move_with_history_and_tt(
    pos: &mut Position,
    game_history: &[ZobristKey],
    limits: &SearchLimits,
    ctx: &SearchContext,
    tt: &mut TranspositionTable,
) -> Option<SearchOutcome>

pub struct SearchLimits { pub depth: Option<u32>, pub nodes: Option<u64> }
pub struct SearchContext { pub stop: Arc<AtomicBool>, /* … */ pub nodes: AtomicU64 }

pub struct SearchOutcome {
    pub best_move: Move,
    pub score: Option<i32>,
    pub completed_depth: u32,
    pub stopped: bool,
    pub pv: Vec<Move>,
}
```

Node count is read from `ctx.nodes.load(Ordering::Relaxed)` after the search
(`SearchOutcome` carries no node field). `MATE = 1_000_000` and
`MATE_THRESHOLD = 999_000` (`crate::engine::tt::MATE_THRESHOLD`) mark
mate scores.

## 6. TT modes

### 6.1 disabled

Use `TranspositionTable::disabled()`. Recreated per fixture, per repeat.

Purpose: exact search baseline; stays consistent with the locked regression;
measures fixed-depth tree size when later ordering / search changes.

### 6.2 cold

Per fixture, per repeat, create a fresh enabled TT (16 MB). Then measure
once. TT is **not** shared across fixtures.

Purpose: simulates a newly allocated or just-cleared production TT; measures the
first search.

### 6.3 warm

For one fixture:

1. create a fresh 16 MB TT;
2. run one warm-up search **not** counted in the result summary;
3. keep that TT;
4. rebuild `Position`, `SearchContext`, and history;
5. run one measured search on the same TT.

Requirements:

- warm-up and measured use the same FEN, history, and limit;
- `Position` must not carry warm-up mutable state into the measured run;
- TT is not shared across fixtures;
- warm-up nodes / time are not counted in aggregate;
- the measured result is printed separately.

## 7. Benchmark suites

### 7.1 Smoke suite

Goal: closeout and daily manual checks run fast.

Must contain the two locked fixtures:

#### startpos disabled depth 3

```text
nodes = 1149
score = 50
bestmove = b1c3
PV = [b1c3,b8c6,g1f3]
```

#### queen-win disabled depth 3

Reuse the currently locked queen-win FEN (do **not** copy an approximate
position):

```text
FEN  = 7k/8/8/8/q3Q2p/8/8/4K3 w - - 0 1
nodes = 963
score = 890
bestmove = e4a4
PV = [e4a4,h4h3,a4h4,h8g8,h4h3]
```

Smoke must:

- use fixed depth;
- verify all exact fields;
- verify the PV is legal step by step;
- verify the position is fully restored after search;
- be designed for fast completion under `--release`;
- not contain a full endgame playout.

### 7.2 Standard fixed-depth suite

8–12 single-position fixtures, covering at least:

1. startpos;
2. locked queen-win;
3. open tactical middlegame;
4. quiet / closed middlegame;
5. exposed-king / attacking position;
6. high-branching movegen position;
7. rook-pawn endgame;
8. KQK low-branching endgame;
9. KRK low-branching endgame;
10. halfmove / repetition context-sensitive position.

Requirements:

- each fixture has a stable, unique short ID;
- each fixture records its FEN;
- history-aware fixtures record a full legal history;
- depth chosen by position cost, typically 3–5;
- the whole suite should be suitable for explicit manual runs, not tens of minutes;
- KQK / KRK here only measure single-position search speed, not a forced mate;
- a correct bestmove is *not* universally required to equal some unproven-unique
  move.

Each fixture documents which search shape it covers.

History convention for M4.0: every fixture uses a single-key history
`[pos.zobrist_key()]` (the position reached by zero plies). This satisfies the
search contract (`game_history` non-empty, last element equals the root
Zobrist key) and keeps the baseline deterministic. The halfmove component of
the TT key is exercised via the fixture FEN (e.g. a high `halfmove_clock`),
which is what the "halfmove / repetition context" fixture targets.

### 7.3 Throughput suite

Fixed-node NPS measurement.

Requirements:

- choose several non-terminal, normally-branching fixtures;
- default node budget example: `100000`;
- the node budget must be honored exactly under existing `SearchContext` semantics;
- a search may stop mid-iteration;
- `stopped == true` is not automatically a failure;
- `Position` and history restoration must be guaranteed;
- under fixed nodes, bestmove / completed depth are **not** treated as a stable
  correctness baseline;
- report elapsed and NPS primarily.

At minimum verify:

```text

for a fixed-depth complete search:

    completed_depth == requested depth

```

Terminal root positions (no legal move) are excluded from the standard suite
unless they are a dedicated fixture.

## 9. Output format

Existing search prints UCI `info` lines. Do **not** refactor search output
to silence it for benchmarking. The benchmark's own output uses easy-to-filter
fixed prefixes:

```text
bench_result
bench_summary
bench_error
```

Suggested per-result format:

```text
bench_result suite=standard fixture=startpos mode=disabled repeat=1 limit=depth:4 score=cp:0 bestmove=b1c3 completed_depth=4 stopped=false nodes=8453 elapsed_us=413000 nps=20467 pv="b1c3 b8c6 g1f3 g8f6"
```

Requirements:

- one result per line;
- fixed key order;
- integer values;
- PV uses UCI moves inside quotes;
- no PV → `pv=""`;
- mate score printed as `mate:N`; ordinary score as `cp:N`;
- no locale dependence;
- no ANSI color;
- extractable via:

```bash
cargo run --release -- bench standard 2>&1 | grep '^bench_'
```

Summary provides at least:

```text
fixture count
measured run count
aggregate nodes
median elapsed_us
median NPS
mode
suite
```

If `repeat > 1`:

- elapsed and NPS use the median;
- deterministic fields (nodes / score / bestmove / depth) are checked for
  consistency across repeats;
- on inconsistency, print a clear warning or error — never silently average.

## 10. Correctness vs performance

### 10.1 Locked correctness fields

Only for fixtures with an explicit contract:

```text
score
bestmove
PV
nodes
completed depth
```

i.e. the smoke startpos and queen-win at disabled depth 3.

### 10.2 Mutable but recorded fields

Later search optimizations may legitimately change:

```text
nodes
bestmove
PV
```

(e.g. a different root move with equal score, or a hash-move tie-break shift).
Therefore the standard suite must NOT hard-freeze every current result into a
permanent unit test. What stays locked:

- score semantics;
- bestmove legality;
- PV legality;
- terminal / draw / mate semantics;
- `Position` / `SearchPath` restoration.

### 10.3 Machine-dependent fields

These must NOT be hard unit-test assertions:

```text
elapsed
NPS
CPU usage
completed depth under node / time limit
```

They enter only the baseline report and later comparisons.

- fixture IDs unique;
- all fixture FENs parse;

- CLI legal-argument parsing;
- CLI illegal-argument returns an error and does not panic;
- mode parsing;
- `repeat` must be > 0;
- `nodes` must be > 0;
- result-line format is stable;
- median computation is correct;
- score formatting is correct;

Forbidden in ordinary tests:

- full standard suite;
- full throughput 100000-node suite;
- full warm benchmark;
- full endgame playout;
- wall-clock performance assertions.

After adding benchmark tests, measure and report the change in total
`cargo test` / `cargo test --release` runtime. Measure and report only; no
hard assertion.

### 21.6 Known measurement limits / P2

- **Single machine, single run.** Numbers are reproducible on this host but are
  not a statistically averaged cloud result; they are a *reference*, not a
  guarantee of cross-machine ratios.
- **Fixture history is single-key** `[pos.zobrist_key()]` by design (spec
  §7.2). The halfmove component of the TT key is exercised via the fixture FEN
  (e.g. `halfmove-ctx` has `halfmove=80`); true *repetition* context (prior
  distinct positions) is not built into the fixtures. Deterministic and
  sufficient for a baseline.
- **Standard-suite results are not frozen as unit tests** (spec §10.2): later
  optimizations may legitimately change `nodes`/`bestmove`/`PV`. What stays
  locked: score semantics, bestmove/PV legality, terminal/draw/mate semantics,
  and `Position`/`SearchPath` restoration.
- **Machine-dependent fields (elapsed/NPS) are never hard unit-test
  assertions** (spec §10.3).
- **M3.2 P2 doc/comment cleanup** (the residual comment + README wording items)
  is tracked separately and explicitly out of scope for M4.0 — not folded into
  any M4.0 commit.

## 22. Remote-review hardening (post-closeout fix)

The first closeout `fef4045` was **REJECTED** on review with **1×P1**: the
fixed-depth completeness invariants (`stopped == false`, `completed_depth ==
requested depth`, `score.is_some()`) were gated behind `mode ==
BenchMode::Disabled`, so a partial `cold`/`warm` fixed-depth result would have
been accepted into the baseline. A new hardening commit
`fix(bench): validate fixed-depth measurements` was added **on top of**
`fef4045` (no amend / no rewrite of the four original commits).

### 22.1 P1 fix (bench.rs)

- A shared `check_fixed_depth_complete(fx, label, out, d)` helper now enforces
  the three invariants for **every** fixed-depth run regardless of mode
  (disabled / cold / warm). `validate()` calls it for any `LimitKind::Depth(d)`.
- The locked *exact* assertions (exact nodes / score / bestmove / PV) remain
  disabled-mode-only, as before — they still apply only to the two locked
  smoke fixtures (disabled depth 3).
- `run_warmup()` validates the warm-up search before the measured run reuses
  the TT: for a fixed-depth warm-up it requires `Some`, `!stopped`,
  `completed_depth == requested depth`, and `score.is_some()`; for a
  node-budget (throughput) warm-up it requires `Some` + `score.is_some()`
  and does **not** demand a specific depth (mid-iteration node stop is normal).
  If the warm-up fails, `run_one` returns a clear error and the measured
  run never executes on a half-warmed TT.

### 22.2 P1 fast tests (no default-gate slowdown)

  `score.is_some()`, bestmove present, PV legal replay.

  warm-up then a measured run; both reach depth 1, `!stopped`,
  `score.is_some()`, bestmove present, PV legal replay.
