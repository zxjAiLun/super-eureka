## 0. 目标与不变式

在现有 material-only 评估（`src/engine/eval.rs` 的 `pub fn evaluate(pos: &Position) -> i32`）之上，**叠加基础 Piece-Square Table（PST）位置奖励**，让引擎开始理解发展、中心、兵的推进位置、车/象/后的活跃格等基础棋理。

三条硬性不变式（本里程碑最重要的验收点）：

1. **`evaluate` 签名与视角不变。** 仍是 `pub fn evaluate(pos: &Position) -> i32`，返回**走子方视角**（本方加分、对方减分）。所有 M2.1/M2.2/M2.3 调用点、`SearchOutcome`、UCI 层零改动。
2. **PST 是纯加分项，叠加在 material 之上。** 不改变 `PieceType::value()`（P=100/N=320/B=330/R=500/Q=900/K=20000）；不改变走子方视角语义。
3. **`evaluate` 不修改 `Position`。** 不 `make_move` / `unmake_move`、不写 `board`、不写任何可变状态；可安全在只读 `&Position` 上重复调用。

> 与 M2.3 的关键区别：**M2.3 要求 PV 不改变任何节点数**（PV 只是记录）；**M2.4 不要求节点数不变** —— 评估函数变化会改变 alpha-beta 的 cutoff 决策，这是预期且正确的（更聪明的评估 → 不同的剪枝）。因此 §7 明确**不**锁定 610/927 旧基线，而是把新的固定基线**归属到新的 M2.4 测试**，并从 M2.3 测试里移除/改造那两个已过时的 exact-node 断言（见 §7）。

## 1. PST 存储与查表

### 1.1 表结构与查表函数

五张 `[i32; 64]` 非王表 + 一张全 0 King 表，**完整数值在附录 §11 锁定**（不再由实现者自选）。索引即 square index（`a1 = 0` … `h8 = 63`）。

查表**必须用穷尽 `match`**，不要用 `PST[pt as usize]`：后者静默依赖 `PieceType` 的枚举声明顺序，未来有人调整 enum 顺序时代码仍能编译却查错表；穷尽 `match` 才能让编译器强制“新增棋子类型必须显式处理”。改动仅在 `eval.rs` 内，不扩大任何接口。

```rust
fn pst_for(pt: PieceType) -> &'static [i32; 64] {
    match pt {
        PieceType::Pawn => &PAWN_PST,
        PieceType::Knight => &KNIGHT_PST,
        PieceType::Bishop => &BISHOP_PST,
        PieceType::Rook => &ROOK_PST,
        PieceType::Queen => &QUEEN_PST,
        PieceType::King => &KING_PST, // KING_PST = [0; 64]，恒贡献 0
    }
}
```

- 表数值来源：经典 Tomasz Michniewski “Simplified Evaluation Function”（chessprogramming wiki）。**原表以 a8..h8 起始（rank 8 在最前），附录 §11 已整体按 rank 反转为 rank 1 在最前**，以匹配本项目 `a1 = 0` 的数组约定（见 §11 的排列说明）。
- 本里程碑**不做调参 / 自对弈优化**（见 §6）；数值量级明显小于对应子力值（单格 ≤ 50），保证 §5 测试 4 的“子力主导”成立。

### 1.2 颜色镜像（黑方复用同一张表）

square 约定（`src/chess/types.rs`）：`a1 = 0`，`h1 = 7`，`a8 = 56`，`h8 = 63`；`file = sq & 7`，`rank = sq >> 3`。

黑方棋子查表前把 square 做**垂直镜像**（rank r ↔ 7−r，file 不变）：

```rust
fn pst_idx(sq: usize, color: Color) -> usize {
    if color == Color::Black { sq ^ 56 } else { sq }
}
```

`sq ^ 56`（`56 = 0b111000`）正好翻转 rank 的三位：`c3`(18) ↔ `c6`(42)、`a1`(0) ↔ `a8`(56)。**不**对 file 做镜像（PST 的左右对称已直接体现在表数据本身，黑方只需翻 rank）。

### 1.3 累加方式

遍历 `pos.board[0..64]`，对每个 `Some(p)`：

```rust
let idx = pst_idx(sq, p.color);
let bonus = pst_for(p.piece_type)[idx];
if p.color == pos.side {
    score += bonus;          // 本方：加位置奖励
} else {
    score -= bonus;          // 对方：减位置奖励（idx 已是镜像后的格）
}
score += p.piece_type.value() * if p.color == pos.side { 1 } else { -1 };
```

- 王（`PieceType::King`）：`KING_PST` 全 0，其 `bonus` 恒为 0，故王只贡献 material，不贡献位置项。§6 明确 King PST 留到 tapered eval。
- 双方各恰好一王，`±20000` 的王 material 在对称王局面下相消（§5 多个纯 evaluate 单测据此得到精确期望值）。
- 最终 `score` 即走子方视角的 `material + positional`，满足 §0 不变式 2。

### 1.4 表数据放在哪

与现有 `evaluate` 同文件 `src/engine/eval.rs`（保持小而内聚；不新建模块、不引入 crate）。同时把 `eval.rs` 顶部 `Planned additions` 注释里的 `piece-square tables` 一项标为已完成（实现提交里顺手改）。

## 2. 数值约束（由附录 §11 锁定 + §5 测试引用具体值）

数值不再“实现者自选”，全部由附录 §11 写死；§5 测试断言直接引用附录的具体格值。以下派生性质由锁定表保证（实现时若与下列不符，说明表贴反/抄错）：

- **颜色镜像（由非对称 Pawn 对真正锁死）**：`PAWN_PST[d7] == 50` 而 `PAWN_PST[d2] == -20`，二者**不等**；故白兵 d7（`k7/3P4/8/8/8/8/8/4K3 w`，eval=150）与黑兵 d2 经 `d2 ^ 56 == d7` 镜像后（eval 同为 150）才是 `sq ^ 56` 的有效证明。注意 `KNIGHT_PST[c3] == KNIGHT_PST[c6] == 10` 是**巧合对称**，**不能**证明镜像（漏掉 `sq ^ 56` 时黑马直接读 c6 仍得 10，测试照样过）；Knight `c3`/`c6` 仅作补充 sanity，见 §5 测试 1。
- **中心优于边缘**：`KNIGHT_PST[c3] (10) > KNIGHT_PST[a3] (-30)`。
- **PST 改变选择（§5 测试 7）**：`KNIGHT_PST[g3] (5) > KNIGHT_PST[f2] (0)`。
- **子力主导**：`QUEEN_PST[a1] == -20`，`eval(白多一后 a1) = 900 + (-20) = 880 > 0`，绝不被 PST 反转。
- **Pawn 垂直非对称（§5 测试 5）**：`PAWN_PST[d7] == 50`，其垂直镜像格 `PAWN_PST[d2] == -20`；两者差 70，可判别“整表被上下贴反”。

## 5. 必须测试

> **通用硬约束（所有涉及搜索的测试）：** 必须显式传入 `depth`，例如
> `let limits = SearchLimits { depth: Some(1), ..Default::default() };`。
> **绝不可**依赖 `SearchLimits::default()`——它是 `depth=None, nodes=None`、无 deadline，语义为“无限迭代直到外部 stop”，测试会挂死。

纯 `evaluate` 单测可放 `eval.rs` 私有测试或 `tests/m2_4.rs`，但需能从 FEN 构造 `Position`。

1. **颜色镜像对称（主证明 = 非对称 Pawn 镜像对，锁死 `sq ^ 56`）**：白兵 `d7` 与黑兵 `d2` 必须都得到 `150`，且两者相等。Knight `c3`/`c6` 因表中巧合对称（都 `== 10`）**不能**证明镜像，仅作补充 sanity（见下）。
   - 白兵 d7：`k7/3P4/8/8/8/8/8/4K3 w - - 0 1`（白 P d7 + 白 K e1 + 黑 K a8，White 走子；d7 兵攻击 c8/e8，黑王在 a8 不被将，合法）
     `evaluate == 150`（`= 100 + PAWN_PST[d7](50)`；双王 ±20000 相消）。
   - 黑兵 d2（颜色互换 + 垂直镜像）：`4k3/8/8/8/8/8/3p4/K7 b - - 0 1`（黑 p d2 + 白 K a1 + 黑 K e8，Black 走子）
     正确实现必须 `d2 ^ 56 == d7` → `evaluate == 150`（`= 100 + PAWN_PST[d7](50)`；双王 ±20000 相消）。**若漏掉黑方镜像**，会直接读 `PAWN_PST[d2] == -20` → 结果 `80`，测试正确失败。
   断言（建议）：
   ```rust
   assert_eq!(evaluate(&white_d7), 150);
   assert_eq!(evaluate(&black_d2), 150);
   assert_eq!(evaluate(&white_d7), evaluate(&black_d2));
   ```
   - **补充 sanity（Knight `c3`/`c6`，非主证明）**：`4k3/8/8/8/8/2N5/8/4K3 w - - 0 1` 与 `4K3/8/2n5/8/8/8/8/4k3 b - - 0 1`，断言 `evaluate(white_N_c3) == evaluate(black_N_c6)`（两者都查 `KNIGHT_PST[c3] == 10`：白 N 直接查 `c3`，黑 N 查镜像 `c6 ^ 56 == c3`）。此条在漏掉 `sq ^ 56` 时仍会通过（表中 `c3 == c6 == 10`），故仅作对称性 sanity，不构成镜像证明。

2. **走子方视角反号**：同一棋盘只切 `side` 字段，评估值应取反。
   - `4k3/8/8/8/8/2N5/8/4K3 w - - 0 1`（White 走子）
   - `4k3/8/8/8/8/2N5/8/4K3 b - - 0 1`（同盘，仅切 side 为 Black）
   断言 `evaluate(white_to_move) == -evaluate(black_to_move)`（白走子 = 320 + 10，黑走子 = -(320 + 10)）。

3. **中心优于边缘**：`evaluate(white_N_c3) - evaluate(white_N_a3) > 0`，等价于 `KNIGHT_PST[c3] (10) > KNIGHT_PST[a3] (-30)`。
   - `4k3/8/8/8/8/2N5/8/4K3 w - - 0 1`（白 N c3）
   - `4k3/8/8/8/8/N7/8/4K3 w - - 0 1`（白 N a3；白 K e1、黑 K e8）

4. **纯子力仍占主导**：白方多一后（后置于最坏角落 `a1`）vs 无后。
   - `4k3/8/8/8/8/8/8/Q3K3 w - - 0 1`（白 Q a1 + 白 K e1 + 黑 K e8；`eval = 900 + QUEEN_PST[a1](-20) = 880 > 0`）
   - `4k3/8/8/8/8/8/8/4K3 w - - 0 1`（同盘去掉后；`eval = 0`）
   断言 `evaluate(queen_up) == 880`、`evaluate(no_queen) == 0`、且 `queen_up > no_queen`。

5. **Pawn 固定值 + 垂直翻转判别（新增，防整表上下贴反）**：Knight 表近似上下对称，无法暴露“整表被垂直镜像”的错误；Pawn 表强不对称，用它锁死方向。
   - `k7/3P4/8/8/8/8/8/4K3 w - - 0 1`（白 P d7 + 白 K e1 + 黑 K a8，White 走子；d7 兵攻击 c8/e8，黑王在 a8 不被将，合法）
   断言 `evaluate(pos) == 150`（= 兵 material 100 + `PAWN_PST[d7] == 50`；双王 ±20000 相消）。
   **判别力**：若整表被上下贴反，`d7` 会读到 `d2` 的 `-20`，结果变成 `80`；本断言即失败。

6. **`evaluate` 不修改 `Position`**：调用前后 `to_fen(&pos)`（或 `pos.board` 逐字节）完全一致。

7. **PST 真正改变选择（同一局面：M2.3 基线 → M2.4 结果）**：构造**无任何吃子、马的 PST 最优落点严格压过所有其它着法**的局面，断言搜索首着从纯 material 基线切换到 PST 更优格。**必须显式 `depth: Some(1)`。**
   - FEN：`4k3/8/8/8/8/8/K7/7N w - - 0 1`（白 N h1 + 白 K a2 + 黑 K e8）
   - **M2.3（6e48606）实测基线**：`best_move == h1f2`（本 spec 已用 `6e48606` 二进制实跑确认；material 全相等 → tie-break 取首个生成着法；`KNIGHT_OFFSETS` 对 h1 先生成 `f2`(`(-2,1)`) 再 `g3`(`(-1,2)`)，且白王在 a2、square index 8 > 马 h1 的 7，不抢占 tie-break）。

## 8. 验收与提交

固定深度、无中断时：

## 11. 附录：锁定的 PST 常量（a1-first）

**排列约定（最重要）：** 数组下标 = square index。`index 0 = a1`，`index 7 = h1`，`index 56 = a8`，`index 63 = h8`。每 8 个一行：**第 1 行 = a1..h1（rank 1），第 2 行 = a2..h2（rank 2），…，第 8 行 = a8..h8（rank 8）**。
