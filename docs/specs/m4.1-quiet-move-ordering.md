# M4.1 — Quiet Move Ordering (Killer Moves + History Heuristic)

**Status:** DRAFT
**Base:** `1b7d7ab` (M4.0 closeout, `fix(bench): validate fixed-depth measurements`)
**Goal:** Improve move ordering for *quiet* moves via Killer Moves and the
History Heuristic so that alpha-beta produces cutoffs earlier, reducing
the number of nodes searched at a fixed depth. **No selective pruning,
no depth change, no evaluation change.**

This milestone is **independent** of M4.0 and of every later search
milestone. It MUST be specified, accepted, and closed out on its own.
No other search algorithm (PVS, aspiration, SEE, LMR, null move,
futility, move-list rewrite, incremental eval, bitboard, multithread,
tapered/mop-up eval, Syzygy) may be bundled into this milestone's
commits or its Goal.

## 3. Killer Moves

### 3.1 Data structure & lifecycle
A single search-local `SearchHeuristics` struct (no global mutable state)
is created **once at the start of each `search_best_move` call** and
carried through **all iterative-deepening iterations of that call**. It is
**re-zeroed on the next independent `search` / `go` call** — never
persisted across `go` searches, across games, or into quiescence.

```rust
// created at the start of one search_best_move call
struct SearchHeuristics {
    // killers[ply] = [slot0, slot1]; grows on demand, see ensure_ply
    killers: Vec<[Option<Move>; 2]>,
}

impl SearchHeuristics {
    fn new() -> Self { SearchHeuristics { killers: Vec::new() } }

    // ply has NO fixed upper bound: `go depth N` accepts any parseable
    // u32, and depth-only / nodes-only / time-only / infinite / bare
    // `go` all leave SearchLimits.depth = None at times; iterative
    // deepening keeps increasing depth until budget/deadline/stop.
    // We therefore grow the table lazily instead of sizing it up front.
    fn ensure_ply(&mut self, ply: usize) {
        if self.killers.len() <= ply {
            self.killers.resize(ply + 1, [None, None]);
        }
    }
}
```
Indexed by `ply` (root ply = 0). Each ply holds two slots
`[slot0, slot1]`. `ensure_ply(ply)` MUST be called (and the table
grown if needed) before any read/write at `ply`.

**Reference profile:** `M4Reference` MUST NOT create, read, or update any
heuristic state. To preserve the historical baseline with no unnecessary
per-node ordering/update overhead, the Reference path skips
`SearchHeuristics` entirely (killer/history ordering is simply absent,
as in M4.0).

### 3.2 Update rules (ALL must hold)
- A killer is recorded **only after a real quiet beta-cutoff**:
  the move `m` is quiet (NOT capture, NOT en passant, NOT promotion),
  and `alpha >= beta` was produced by `m` at this node (a true
  cutoff, not a TT/standing-pat return).
- The update is applied **only at non-root ordinary negamax nodes**
  (see §5 root scope). It is **never** applied inside quiescence.
- The locked update (called after a qualifying cutoff with move `m` at
  `ply`):
  ```rust
  let k = &mut self.killers[ply];
  if k[0] != Some(m) {   // m == slot0 -> no-op (no duplicate)
      k[1] = k[0];       // old slot0 demoted to slot1
      k[0] = Some(m);     // m becomes slot0
  }
  // result: slot0 and slot1 are never equal
  ```
- Behaviors that this rule guarantees:
  - `m == slot0` → **no-op** (no duplicate created);
  - `m == slot1` → promoted to `slot0`, old `slot0` demoted to
    `slot1` (slots still distinct);
  - brand-new `m` → inserted at `slot0`, old `slot0` demoted to
    `slot1`;
  - after every update, `slot0 != slot1` is invariant.
- Capture / en passant / promotion moves MUST NOT become killers.
- Lifecycle: `SearchHeuristics` is created at the start of the
  `search_best_move` call, retained across all iterative-deepening
  iterations of that call, and re-zeroed on the next independent
  `search` / `go` call. It is never persisted across `go` searches,
  across games, or into quiescence.

### 3.3 Usage rules (ALL must hold)
- When ordering a node's moves, a killer slot is placed only if that
  move is present in the node's current **legal** move list and has not
  already been placed (e.g., it is not the TT hash move, not already
  emitted).
- Killers are ordered: slot0 before slot1 (see §5, levels 4–5).
- Killers are **never** used inside quiescence.
- Killers are used **only at non-root ordinary negamax nodes**. The root
  node keeps the M4.0 ordering logic for both profiles (see §5).

## 4. History Heuristic

### 4.1 Table definition
```rust
// per independent search; reset to zero at every search entry
const M4_HISTORY_CAP: i32 = 16_384;
let mut history: [[[i32; 64]; 64]; 2] = [[[0; 64]; 64]; 2];
// index: history[color][from][to], color = side to move at the node
```

### 4.2 Update formula (locked)
- On a real quiet beta-cutoff move `m` at a node whose remaining
  search depth is `d` (the depth at which `m` is played):
  ```rust
  // lossless widening first; NO reliance on any "depth <= 64" bound
  let d = u64::from(d);                                 // d: u32 depth, widened
  let bonus = d
      .saturating_mul(d)                               // d*d, never overflows u64
      .min(M4_HISTORY_CAP as u64) as i32;             // cap before i32 cast
  let color = pos.side_to_move() as usize;              // mover's color
  let idx = m.from as usize;
  let jdx = m.to   as usize;
  let updated = history[color][idx][jdx]
      .saturating_add(bonus)
      .min(M4_HISTORY_CAP);                            // bounded normalization
  history[color][idx][jdx] = updated;
  ```
- Reward grows with search depth (`d*d`): a cutoff nearer the root
  (larger `d`) contributes more.
- **No integer overflow is possible for ANY legal `u32` depth**:
  `d` is first widened losslessly to `u64` (so `u32::MAX` does NOT
  truncate or wrap on the cast); `d*d` uses `saturating_mul` and is
  capped at `M4_HISTORY_CAP as u64` *before* the `as i32` cast; the
  running sum uses `saturating_add` and is capped at
  `M4_HISTORY_CAP`. All arithmetic is overflow-free even though no
  `depth <= 64` assumption is made. (In practice depth never reaches
  such values, but the formula does not depend on that.)

### 4.3 Normalization policy (locked)
- v1 keeps the table **fresh per search** and **caps every entry** at
  `M4_HISTORY_CAP`. This cap is the explicit **bounded normalization**
  strategy (NOT a periodic decay): it bounds table growth and prevents
  overflow within a single search. No separate periodic decay or
  division is performed in v1 (this keeps ordering deterministic and
  trivially overflow-free).
- Lifecycle: the table is created (zeroed) at the start of the
  `search_best_move` call, retained across all iterative-deepening
  iterations of that call, and re-zeroed on the next independent
  `search` / `go` call. It is **not** persisted across games or
  independent `go` searches, and **never** read or updated inside
  quiescence. `M4Reference` does not create, read, or update the
  history table.

### 4.4 Scope (ALL must hold)
- History is consulted and updated **only for quiet moves** (category 0;
  see §5). Captures, en passant, and promotions keep their existing
  MVV-LVA / promotion ranking and are **never** history-updated.
- History is **never** used in quiescence.
- The History Heuristic table is distinct from the search-line
  `SearchPath` "history" (the Zobrist-key stack). The two names must
  not be conflated in code or comments.

## 5. Formal Move Ordering Priority

Within a node, every legal move appears **exactly once**, in this order:

1. **Legal TT hash move** (lifted to index 0, as today).
2. **Promotions** (existing category 2, MVV-LVA within promotions).
3. **Captures + en passant**, existing strict MVV-LVA
   (most valuable victim first; cheapest attacker first; en passant valued
   as a pawn).
4. **Killer slot 0** (quiet only; only if legal & present & not
   already emitted).
5. **Killer slot 1** (quiet only; same guards).
6. **Remaining quiet moves**, sorted by
   `history[color][from][to]` **descending**.
7. **Deterministic tie-break** when history values are equal: order by
   `(from, to)` ascending, then by generation index ascending (matches
   the existing stable `order_moves` tie-break).

Integration note: the existing `move_order_key` already produces
categories 1–3 (promotion / capture / quiet). M4.1 only **subdivides
the quiet band (category 0)** into killers (levels 4–5) then
history-sorted quiets (level 6) with the deterministic tie-break
(level 7). Categories 1–3 and the TT-hash-first rule (level 1) are
left exactly as M4.0. The composite ordering key therefore extends the
quiet band's sub-rank only; it never reorders captures/promotions
relative to each other or to the TT move.

### 5.1 Root node & qsearch scope (locked)
- **Reference root (`M4Reference`):** the root ordering logic is
  **exactly** the M4.0 logic — TT hash move lifted to index 0,
  remaining root moves in their existing order; no full MVV-LVA /
  promotion / history re-sort. Behavior is byte-identical to M4.0.
- **Current root (`Current`):** the root ordering ALSO keeps the
  M4.0 TT-hash-move + previous-iteration-best mechanism. The seven-
  level killer/history ordering (§5 levels 4–7) is applied
  **ONLY to non-root ordinary negamax nodes**, NOT to the root. This
  is the narrowest interpretation of "only improve quiet beta-cutoff
  ordering" and keeps the historical root baseline intact.
- **qsearch:** the quiescence move ordering is **fully unchanged** for
  both profiles — no killer, no history, no seven-level ordering.
- If a future change wants killer/history ordering at the root, it
  MUST be a separate, independently-scoped milestone; it is NOT in
  M4.1's scope and MUST NOT be decided by the implementer.

## 6. Correctness Acceptance (M4.0Reference vs Current)

On the M4.0 **standard** fixture set, for every fixture, a fixed-depth
**complete** search under `Current` MUST satisfy, relative to
`M4Reference` at the same depth:

- **score identical** (including sign/magnitude of cp and mate scores);
- **mate distance identical**;
- **draw-claim semantics identical** (insufficient-material auto-draw;
  fifty-move / threefold claimable-0 options; current/intended-move
  claim; terminal priority);
- **terminal priority unchanged**;
- **bestmove legal**;
- **PV stepwise legal** (each PV move is legal in its preceding
  position);
- **Position / FEN / Zobrist / SearchPath fully restored** after the
  search (root position byte-identical; search-line history residue
  absent);
- **after abort, the last fully-completed iteration is returned**
  (`score: None, stopped: true` only at the top level; lower-level
  aborts return `None` with board intact);
- **TT bound and best-move semantics unchanged** (exact/lower/upper
  classification, depth-preferred replacement, mate-ply codec, legal
  hash-move ordering).

The "node / stop semantics" constraint is **limit-type dependent** and
is split into three explicit contracts below. It is intentionally NOT a
blanket "identical under equal limits" requirement, because reducing
nodes is exactly what `Current` is for.

#### 6.1 Node / stop correctness by limit type
- **Fixed depth** (`SearchLimits { depth: Some(d), .. }`):
  - both `M4Reference` and `Current` must reach the requested depth
    **completely** (M4.0 fixed-depth completeness: `stopped == false`,
    `completed_depth == d`, `score.is_some()`);
  - **score and mate distance are identical**;
  - **node counts MAY and ARE EXPECTED to differ** (`Current` should
    search fewer nodes) — this is the goal, not a regression.
- **Fixed nodes** (`SearchLimits { nodes: Some(budget), .. }`):
  - nodes MUST NOT exceed the budget;
  - when the budget is exhausted, **actual nodes == requested budget**;
  - `completed_depth` / `bestmove` / `PV` MAY differ between profiles
    (M4.0 already states node-limit completed-depth/bestmove are not a
    stable correctness baseline);
  - the last fully-completed iteration and restoration semantics MUST
    match.
- **Fixed time / external stop** (time-only, `go infinite`, bare
  `go`, or an externally-set `stop` flag):
  - node count and `completed_depth` are NOT required to match between
    profiles (thread scheduling and machine speed decide the stop point);
  - both MUST stop promptly, return the last fully-completed iteration,
    perform NO partial TT store, and fully restore state.

`bestmove` and `PV` **MAY differ** between `M4Reference` and
`Current` when multiple moves share the same score (reordering of
equal-score moves is expected and correct). Only the items above are
hard constraints.

## 7. Performance Acceptance

Reference (M4.0 disabled aggregate, from
`docs/benchmarks/m4.0-search-baseline.md`): **498,474 nodes**
(measured under `M4Reference`, `--mode disabled`).

`Current` (M4.1, `--mode disabled`) MUST:
- have an aggregate node count **strictly below** the 498,474 reference;
- achieve a **minimum 5% reduction** (aggregate);
- **target 15% or more** reduction (aggregate);
- show **clear improvement on heavy positions** (e.g., `open-tactical`,
  `exposed-king`, `halfmove-ctx`);
- if **any single fixture regresses by more than 20%**, the regression
  MUST be explained in the closeout report (not silently accepted);
- keep the **fixed-node NPS drop ≤ 15%** unless total wall time still
  improves significantly (killer/history bookkeeping has a per-node
  cost; the node reduction must pay for it);
- treat **warm-TT results as recorded-only** — they are NOT the primary
  proof of killer/history effect (TT reuse dominates there).

**Hard gate:** if the total node reduction is **below 5%**, M4.1 MUST
NOT be closed as "done" — it must be revised or explicitly failed, never
closed on a "function complete" pretext.
