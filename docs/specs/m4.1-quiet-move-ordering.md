# M4.1 — Quiet Move Ordering (Killer Moves + History Heuristic)

**Status:** DRAFT
**Base:** `1b7d7ab` (M4.0 closeout, `fix(bench): validate fixed-depth measurements`)
**Goal:** Improve move ordering for *quiet* moves via Killer Moves and the
History Heuristic so that alpha-beta produces cutoffs earlier, reducing
the number of nodes searched at a fixed depth. **No selective pruning,
no depth change, no evaluation change.**

This milestone is **independent** of M4.0 and of every later search
milestone. It MUST be specified, accepted, and closed out on its own.
No other search algorithm (PVS, aspiration, SEE, LMR, null move,
futility, move-list rewrite, incremental eval, bitboard, multithread,
tapered/mop-up eval, Syzygy) may be bundled into this milestone's
commits or its Goal.

## 3. Killer Moves

### 3.1 Data structure
Per independent search, a search-local table (no global mutable state):
```rust
// sized per search: capacity = (max_search_depth + 2)
let mut killers: Vec<[Option<Move>; 2]> = vec![[None, None]; depth as usize + 2];
```
Indexed by `ply` (root ply = 0). Each ply holds two slots
`[slot0, slot1]`.

### 3.2 Update rules (ALL must hold)
- A killer is recorded **only after a real quiet beta-cutoff**:
  the move `m` is quiet (NOT capture, NOT en passant, NOT promotion),
  and `alpha >= beta` was produced by `m` at this node (a true
  cutoff, not a TT/standing-pat return).
- On a qualifying cutoff with move `m`:
  - if `m == slot1` (already present) → do nothing;
  - else → `slot1 = slot0; slot0 = Some(m);`
  - after the shift, `slot0` and `slot1` are never equal (enforced by
    the `m == slot1` guard plus the precondition that `m` just cut and
    was not already `slot0`).
- Capture / en passant / promotion moves MUST NOT become killers.
- The table is created fresh at each search entry and dropped on return.
  It is **not** persisted across independent `go` searches, and **not**
  carried into quiescence.

### 3.3 Usage rules (ALL must hold)
- When ordering a node's moves, a killer slot is placed only if that
  move is present in the node's current **legal** move list and has not
  already been placed (e.g., it is not the TT hash move, not already
  emitted).
- Killers are ordered: slot0 before slot1 (see §5, levels 4–5).
- Killers are **never** used inside quiescence.

## 4. History Heuristic

### 4.1 Table definition
```rust
// per independent search; reset to zero at every search entry
const M4_HISTORY_CAP: i32 = 16_384;
let mut history: [[[i32; 64]; 64]; 2] = [[[0; 64]; 64]; 2];
// index: history[color][from][to], color = side to move at the node
```

### 4.2 Update formula (locked)
- On a real quiet beta-cutoff move `m` at a node whose remaining
  search depth is `d` (the depth at which `m` is played):
  ```
  let bonus = (d as i32).saturating_mul(d as i32);   // d*d, never panics
  let color = pos.side_to_move() as usize;              // mover's color
  let idx = m.from as usize;
  let jdx = m.to   as usize;
  let v = history[color][idx][jdx].saturating_add(bonus);
  history[color][idx][jdx] = v.min(M4_HISTORY_CAP);
  ```
- Reward grows with search depth (`d*d`): a cutoff nearer the root
  (larger `d`) contributes more. `d` is bounded by the search limit
  (≤ 64), so `d*d ≤ 4096`, far inside `i32`.
- **No integer overflow is possible**: `bonus` uses `saturating_mul`;
  the running sum uses `saturating_add` and is capped at
  `M4_HISTORY_CAP`. All arithmetic is `i32`.

### 4.3 Normalization / decay policy (locked)
- v1 keeps the table **fresh per search** and **caps every entry** at
  `M4_HISTORY_CAP`. This cap is the explicit normalization/decay
  strategy: it bounds table growth and prevents overflow within a single
  search. No separate periodic decay or division is performed in v1
  (this keeps ordering deterministic and trivially overflow-free).
- The table is **not** persisted across games or independent `go`
  searches in v1.

### 4.4 Scope (ALL must hold)
- History is consulted and updated **only for quiet moves** (category 0;
  see §5). Captures, en passant, and promotions keep their existing
  MVV-LVA / promotion ranking and are **never** history-updated.
- History is **never** used in quiescence.
- The History Heuristic table is distinct from the search-line
  `SearchPath` "history" (the Zobrist-key stack). The two names must
  not be conflated in code or comments.

## 5. Formal Move Ordering Priority

Within a node, every legal move appears **exactly once**, in this order:

1. **Legal TT hash move** (lifted to index 0, as today).
2. **Promotions** (existing category 2, MVV-LVA within promotions).
3. **Captures + en passant**, existing strict MVV-LVA
   (most valuable victim first; cheapest attacker first; en passant valued
   as a pawn).
4. **Killer slot 0** (quiet only; only if legal & present & not
   already emitted).
5. **Killer slot 1** (quiet only; same guards).
6. **Remaining quiet moves**, sorted by
   `history[color][from][to]` **descending**.
7. **Deterministic tie-break** when history values are equal: order by
   `(from, to)` ascending, then by generation index ascending (matches
   the existing stable `order_moves` tie-break).

Integration note: the existing `move_order_key` already produces
categories 1–3 (promotion / capture / quiet). M4.1 only **subdivides
the quiet band (category 0)** into killers (levels 4–5) then
history-sorted quiets (level 6) with the deterministic tie-break
(level 7). Categories 1–3 and the TT-hash-first rule (level 1) are
left exactly as M4.0. The composite ordering key therefore extends the
quiet band's sub-rank only; it never reorders captures/promotions
relative to each other or to the TT move.

## 6. Correctness Acceptance (M4.0Reference vs Current)

On the M4.0 **standard** fixture set, for every fixture, a fixed-depth
**complete** search under `Current` MUST satisfy, relative to
`M4_0_Reference` at the same depth:

- **score identical** (including sign/magnitude of cp and mate scores);
- **mate distance identical**;
- **draw-claim semantics identical** (insufficient-material auto-draw;
  fifty-move / threefold claimable-0 options; current/intended-move
  claim; terminal priority);
- **terminal priority unchanged**;
- **bestmove legal**;
- **PV stepwise legal** (each PV move is legal in its preceding
  position);
- **Position / FEN / Zobrist / SearchPath fully restored** after the
  search (root position byte-identical; search-line history residue
  absent);
- **node budget and stop semantics unchanged** (`stopped`,
  `completed_depth`, `ctx.nodes` accounting identical under equal
  limits);
- **after abort, the last fully-completed iteration is returned**
  (`score: None, stopped: true` only at the top level; lower-level
  aborts return `None` with board intact);
- **TT bound and best-move semantics unchanged** (exact/lower/upper
  classification, depth-preferred replacement, mate-ply codec, legal
  hash-move ordering).

`bestmove` and `PV` **MAY differ** between `M4_0_Reference` and
`Current` when multiple moves share the same score (reordering of
equal-score moves is expected and correct). Only the items above are
hard constraints.

## 7. Performance Acceptance

Reference (M4.0 disabled aggregate, from
`docs/benchmarks/m4.0-search-baseline.md`): **498,474 nodes**
(measured under `M4_0_Reference`, `--mode disabled`).

`Current` (M4.1, `--mode disabled`) MUST:
- have an aggregate node count **strictly below** the 498,474 reference;
- achieve a **minimum 5% reduction** (aggregate);
- **target 15% or more** reduction (aggregate);
- show **clear improvement on heavy positions** (e.g., `open-tactical`,
  `exposed-king`, `halfmove-ctx`);
- if **any single fixture regresses by more than 20%**, the regression
  MUST be explained in the closeout report (not silently accepted);
- keep the **fixed-node NPS drop ≤ 15%** unless total wall time still
  improves significantly (killer/history bookkeeping has a per-node
  cost; the node reduction must pay for it);
- treat **warm-TT results as recorded-only** — they are NOT the primary
  proof of killer/history effect (TT reuse dominates there).

**Hard gate:** if the total node reduction is **below 5%**, M4.1 MUST
NOT be closed as "done" — it must be revised or explicitly failed, never
closed on a "function complete" pretext.
