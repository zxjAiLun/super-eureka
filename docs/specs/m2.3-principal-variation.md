## 0. 目标与不变式

在现有 alpha-beta + quiescence（M2.1/M2.2）之上，让引擎能输出**完整主变（PV）**，并在 UCI `info` 里逐迭代打印。

两条硬性不变式（本里程碑最重要的验收点）：

1. **PV 不改变任何搜索决策或节点数。** PV 只是对搜索结果的"记录"，绝不触发额外搜索、不改动 move ordering、不改动 alpha/beta 语义、不改动 cutoff 判定、不调用 `try_enter_node`。因此固定深度、无中断时，`best_move` / `score` / `node count` 必须与 M2.2 逐字节一致（见 §8 实测基线）。
2. **部分迭代的 PV 永远不能泄漏成正式结果。** 被 `stop` / `nodes` / `hard deadline` 中止的更深迭代，其未完成的 PV 必须被整体丢弃，不能覆盖上一轮"完整迭代"的结果。

## 1. PV 存储方式

不要让每个 `negamax` / `quiescence` 节点返回一个**新分配的** `Vec<Move>`（热路径大量分配，且强制公开接口与直接调用测试一起改动）。

采用**每轮 root iteration 一个 `PvTable`**（三角 PV 的轻量替代，无 TT）：

```rust
struct PvTable {
    lines: Vec<Vec<Move>>, // 以全局 ply 为索引
}
```

- 节点进入时**清空自己那一行**；发现新 best move 时把"当前着法 + 子节点已记录的主变"并入当前行。
- 保留现有生成顺序与 `>` tie-break（不主动改变 best-move 语义）。
- 不引入新 crate，不引入 TT。

### 1.1 分配语义（非"零分配"）

> 不在每个递归返回值中构造 PV `Vec` —— 这是本设计要消除的分配。
> 但 `PvTable` 行复用已有容量，**行首次增长到超过其当前容量时仍可能发生摊销分配**。这是可接受的，不追求绝对零分配。

不要把 `Vec<Vec<Move>>` 改成复杂的扁平三角数组（flat triangular array）；M2.3 的目标是**正确** PV，不是内存微优化。为避免热路径上的 `clone` / `collect`，所有行写入都通过 `PvTable` 的方法完成，搜索代码里**不要**直接 `clone` 或 `collect` 子行。

### 1.2 PvTable 方法

```rust
impl PvTable {
    /// 只扩不缩：仅当当前行数小于目标时才 resize。
    fn ensure_rows(&mut self, rows: usize) {
        if self.lines.len() < rows {
            self.lines.resize_with(rows, Vec::new);
        }
    }

    /// 确保索引 `index` 所在的行存在（行数 = index + 1）。
    fn ensure_index(&mut self, index: usize) {
        let rows = index
            .checked_add(1)
            .expect("PV table index overflow");
        self.ensure_rows(rows);
    }

    /// 节点成功进入后调用：清空本 ply 行，消除同一迭代内 sibling 残留。
    /// 必须在任何终局 / stand-pat / qply-cap 提前返回之前发生（见 §2.1）。
    fn clear_at(&mut self, ply: usize) {
        self.ensure_index(ply);
        self.lines[ply].clear();
    }

    /// 当前着法 + 子节点行（不 clone 子行，复用 parent 容量）。
    fn set_from_child(&mut self, ply: usize, mv: Move) {
        let child_ply = ply.checked_add(1).expect("PV ply overflow");
        self.ensure_index(child_ply);

        let (parents, children) = self.lines.split_at_mut(child_ply);
        let parent = &mut parents[ply];
        let child = &children[0];

        parent.clear();
        parent.push(mv);
        parent.extend_from_slice(child);
    }

    /// 只写一步（用于 `search_final_evasion_ply` 的一层解将）。
    fn set_single(&mut self, ply: usize, mv: Move) {
        self.ensure_index(ply + 1);
        self.lines[ply].clear();
        self.lines[ply].push(mv);
    }
}
```

### 1.3 容量与清空语义

`root_search` 在**每轮迭代开始前**调用 `self.ensure_rows(rows)`，其中容量公式（用 `usize` 直接 checked 加法，不要先 `u32` 再加）：

```rust
let rows = depth as usize
    .checked_add(MAX_QPLY as usize)
    .and_then(|n| n.checked_add(2))
    .expect("PV table size overflow");
self.ensure_rows(rows);   // 只扩不缩（见 §1.2）
```

- `root_search` 在**每轮迭代开始前**用本轮 `depth` 调用一次 `ensure_rows(rows)`（见上），**绝不**基于 `limits.depth` 一次性分配（异常巨大的 `go depth` 会导致巨大一次性分配）。
- 若当前 root iteration 深度为 `D`：
  - 主搜索 `depth == 0` 节点位于全局 `ply = D`；
  - qsearch 最深进入 `qply = MAX_QPLY` 时位于 `D + MAX_QPLY`；
  - emergency evasion 只在该行记一个实际解将，不再递归。
  - 因此实际需要的**最大行索引**是 `D + MAX_QPLY`。
  - 理论最小行数 = `D + MAX_QPLY + 1`；上面 `+2` 多留一行，安全。
- **跨迭代残留：不存在。** 每轮 root iteration 使用**独立的新 `PvTable`**，所以根本没有跨迭代残留的可能。`clear_at(ply)` 只用于消除**同一迭代内** sibling 搜索留下的旧尾部。未访问的更深行即使有旧内容也**不会被引用**，因为 parent 只复制"刚刚正常返回的 child 行"。
- `clear_at(ply)` 必须在**成功进入节点后、任何终局 / stand-pat / qply-cap 提前返回之前**调用。这样：
  - 终局 child 的 PV 为空；
  - stand-pat child 的 PV 为空；
  - 没有改善的 qsearch child 不会继承 sibling 的尾巴。

### 1.4 公开接口保持不变

```rust
pub fn negamax(...) -> Option<i32>   // 内部建临时 PvTable，调私有 impl 后丢弃 PV
pub fn quiescence(...) -> Option<i32>  // 同上
```

现有直接调用这两个函数的测试（含 `search.rs` 单元测试、`tests/m2_1.rs`、`tests/m2_2.rs`）**签名不变、零改动**。这是本设计刻意保留的约束（现有 M2.1 测试确实直接调用它们）。

### 1.5 私有实现拆分（携带 PvTable）

```rust
fn negamax_impl(..., pv: &mut PvTable) -> Option<i32>
fn quiescence_impl(..., pv: &mut PvTable) -> Option<i32>      // 递归入口，先 try_enter_node 再调 _entered
fn quiescence_entered_impl(..., pv: &mut PvTable) -> Option<i32> // 搜索体
fn search_final_evasion_ply(..., pv: &mut PvTable)             // 现有私有函数，加 pv 参数
```

- `pub fn negamax` → 建 `let mut pv = PvTable::new();` → 调 `negamax_impl(..., &mut pv)` → 返回分值。
- `negamax_impl` 在 `depth == 0` 调 `quiescence_entered_impl(..., pv)`（不重复计数，沿用 `quiescence_entered` 现有语义）；`depth > 0` 递归调 `negamax_impl(pos, depth-1, ply+1, ..., pv)`。
- `pub fn quiescence` → 建临时 `PvTable` → 调 `quiescence_impl(..., &mut pv)`。
- `quiescence_impl` 先 `try_enter_node` 再调 `quiescence_entered_impl`。
- `root_search` **直接持有** `PvTable`，调 `negamax_impl(pos, depth-1, 1, -beta, -alpha, ctx, limits, &mut pv)` 并读取 `pv.lines[0]` 作为完整主变（见 §3）。

## 2. PV 覆盖范围与更新顺序

PV 必须贯穿以下全部路径：

| 路径 | 处理 |
| --- | --- |
| 普通 `negamax` | 每节点 `clear_at(ply)`；某着法成为本节点 best 时 `set_from_child(ply, m)` |
| `quiescence` 战术着法 | 同上；`stand-pat` 不产生后续 PV（该行保持空） |
| 被将时的全部解将 | 全部 legal 解将都搜（`in_check` 分支），best 解将写 `line[ply]` |
| `search_final_evasion_ply` 最终一层解将 | `set_single(ply, m)`（**只记录实际搜过的那一步**） |

**终局 / stalemate / stand-pat / 非将状态的 qply cap** 都产生空的后续 PV（节点进入已 `clear_at`，无 improving move 则行保持空，链式自然终止）。

### 2.1 更新顺序（关键）

> 当某个着法成为当前 best 时，**先更新当前行，再处理可能的 beta cutoff。** 否则 cutoff 着法可能没有进入 PV。

在合法 alpha-beta 窗口中始终有 `alpha < beta`，因此 `score >= beta` ⇒ `score > alpha`。qsearch 没有独立的 `best` 变量，且当前实现是先查 beta cutoff 再查是否提升 alpha。正确规则必须**先写出产生 cutoff 的实际着法**，再返回 `beta`：

**普通 negamax**（有独立 `best` 变量）：

```rust
match child {
    Some(s) => {
        let score = -s;
        pos.unmake_move(undo);
        if score > best {            // 新的本节点 best
            best = score;
            pv.set_from_child(ply, m);  // 先写 PV 行
        }
        if best > alpha { alpha = best; }
        if alpha >= beta { break; }   // cutoff 着法已在上面写入 PV
    }
    None => { pos.unmake_move(undo); return None; }
}
```

**quiescence**（无独立 `best` 变量，必须先记 cutoff 着法）：

```rust
match child {
    Some(s) => {
        let score = -s;
        pos.unmake_move(undo);
        if score >= beta {
            pv.set_from_child(ply, m); // 必须先记录 cutoff move
            return Some(beta);
        }
        if score > alpha {
            pv.set_from_child(ply, m);
            alpha = score;
        }
    }
    None => { pos.unmake_move(undo); return None; }
}
```

**`search_final_evasion_ply`**（直接评分，无递归 child；`score` 已是当前节点视角，不再取负）：

```rust
if !try_enter_node(ctx, limits) {
    return None;
}

let undo = pos.make_move(m);

let child_in_check = pos.is_in_check(pos.side);
let child_legal = generate_legal_moves(pos);
let score = if child_legal.is_empty() {
    if child_in_check {
        MATE - (ply as i32 + 1)
    } else {
        0
    }
} else {
    -evaluate(pos)
};

pos.unmake_move(undo);

if score >= beta {
    pv.set_single(ply, m);   // 必须先记录 cutoff move
    return Some(beta);
}
if score > alpha {
    pv.set_single(ply, m);
    alpha = score;
}
```

- `search_final_evasion_ply` 的 counter-check 静态近似仍在该子局面终止：PV 只用 `set_single` 记实际搜过的那一步，`line[ply+1]` 保持空，**不虚构后续着法**。

## 3. Root 与中断语义

### 3.1 `root_search` 返回完整迭代结果

```rust
struct RootIteration {
    score: i32,
    pv: Vec<Move>,   // 非空；pv[0] == 本轮 best move
}

fn root_search(...) -> Option<RootIteration>   // None = 本轮被中止
```

- 在根着法循环中：当某根着法 `score > best_score` 时，除记录 `best_score` 外，调 `pv.set_from_child(0, m)`（复制 `pv.lines[1]` 进 `pv.lines[0]`）。
- 循环正常结束 → 返回 `Some(RootIteration { score: best_score, pv: pv.lines[0].clone() })`。
- 任一子节点返回 `None`（中止）→ 立即 `unmake` 并 `return None`（整轮作废）。

> **不删除公开 `SearchResult`。** 现有 `enum SearchResult { Stopped, Score(i32) }` 由 `engine::mod` 明确 re-export（公开 API），仓库内虽无调用者，不代表外部使用者不存在。删除它属于公开 API 破坏，与"不 bump 版本"冲突。
> 本轮做法：让 `root_search` 的**内部**返回类型改为 `Option<RootIteration>`；**保留** `SearchResult` 类型定义与 `mod` 的 re-export，即使内部不再使用。下次明确的 API 清理或版本升级再删除。

### 3.2 `search_best_move` 保存"最后完整迭代"

- 维护 `let mut completed: Option<RootIteration> = None;`（best move 可由 `pv[0]` 得到，减少不一致状态）与 `completed_depth: u32`。
- 每轮 `root_search` 返回 `Some(RootIteration)` → 更新 `completed` 与 `completed_depth`，并照旧把 best move `swap(0, idx)` 到 `root_moves` 前部（move-ordering hook 不变）。
- 返回 `None`（中止）→ `stopped = true; break`，**不**触碰 `completed`。
- 构造 `SearchOutcome` 时新增字段：

```rust
pub struct SearchOutcome {
    pub best_move: Move,
    pub score: Option<i32>,
    pub completed_depth: u32,
    pub stopped: bool,
    pub pv: Vec<Move>,   // 新增
}
```

> **`SearchOutcome.pv` 是公开的 API 扩展（有意）。** M2.3 保留现有函数签名与公开 `SearchResult`（避免不必要的 API 删除）；但新增 `pub pv: Vec<Move>` 会让依赖 `SearchOutcome { best_move, score, completed_depth, stopped }` 字面量的外部代码无法编译。项目当前处于 `0.2.x`，新增 PV 输出渠道本就需要该字段，故**接受**这一 source-level API 扩展，即使本里程碑不单独 bump 版本。这与“保留 `SearchResult`”不矛盾：前者是避免*主动删除*公开项，后者是*有意的、已记录的*新增。

PV 语义分三种情况：

| 情况 | `best_move` | `score` | `completed_depth` | `pv` |
| --- | --- | --- | --- | --- |
| 至少完成 depth 1 | 最后完整迭代的 `pv[0]` | 该迭代 score | 该迭代 depth | 最后完整迭代的 `pv`（非空，`pv[0]==best_move`） |
| depth 1 之前立即中止 | 稳定 fallback（`root_moves[0]`，与现有一致） | `None` | `0` | **空**（`completed` 为 `None`，无真实完成的主变） |
| 终局（无合法着法） | — | — | — | `search_best_move` 仍整体返回 `None`（不变） |

**不要把 fallback 伪装成完成 PV**：depth 1 前中止时 `pv` 必须为空。

### 3.3 自检

`search_best_move` 返回前：

```rust
if !outcome.pv.is_empty() {
    debug_assert_eq!(outcome.pv[0], outcome.best_move);
}
```

## 4. UCI 输出

完成每轮迭代后，把现有单行 `pv <root_move>` 扩展为完整 PV：

```text
info depth N score <cp|mate> nodes N time N nps N pv <m1> <m2> <m3> ...
```

- PV 每一步用 `move_to_uci` 转 UCI notation，**必须能从根局面开始逐步合法执行**（由 §5 测试 2 锁住）。
- `bestmove` 仍输出 `SearchOutcome.best_move`（UCI 层逻辑不变）。
- 当 `outcome.pv` 非空时，`bestmove` 与最后完整 `info pv` 的首着一致（由 §5 测试 8 锁住）。
- **本里程碑不重构** `info` 输出为 callback / channel；只在现有 `search_best_move` 内的 println 处（现 `search.rs` 中 `info depth ... pv {}` 那行）把 `move_to_uci(mv)` 换成 `pv.iter().map(move_to_uci).collect::<Vec<_>>().join(" ")`。

## 5. 必须测试

覆盖（既有测试基础设施 `tests/common/mod.rs` 的 `EngineProcess` / `recv_until` 可复用）：

1. 固定深度搜索完成后，`pv` 非空且 `pv[0] == best_move`。
2. 从根 FEN 顺序 `make_move` PV 中每一步，所有着法都合法，最终根局面仍保持不变（字节级还原）。**PV 合法性与根局面恢复是两个不同不变量，分开断言**（见下）。
3. `depth 0` / 叶节点进入 quiescence 时，PV 可以超出 nominal depth。用现有安静升变局面（`e7e8q`），断言 qsearch PV 包含 `e7e8q`。**此测试必须放在 `search.rs` 的私有 unit tests 里**，直接调用 `negamax_impl` / `quiescence_impl` 并检查内部 `PvTable`（公开 `negamax` / `quiescence` 按设计会丢弃 PV，调用者拿不到）。备选：设计一个 depth 1 的公开局面使 qsearch 产生第二步战术续着，断言 `outcome.pv.len() > outcome.completed_depth as usize`；前者更直接、更稳定，优先采用前者。
4. mate-in-one 的 PV 以 `a1a8` 开始，score 仍为 mate。
5. **“上一完整迭代保留、更深迭代中止不泄漏”用确定性 node budget 锁住**（不依赖线程时序）。基于 startpos 基线 d1=20，设 `SearchLimits { depth: None, nodes: Some(21) }`：
   - depth 1 用满前 20 个节点并完成；
   - depth 2 开始后再获得至多 1 个节点，随即因节点预算中止；
   - 返回 `completed_depth == 1`、`stopped == true`、`score` 与 `pv` 来自 depth 1、`pv[0] == best_move`。
   对 `stop`（预设 stop flag）与 `hard deadline`（已过期 deadline）：只锁住“depth 1 之前中止 → `score == None` / `completed_depth == 0` / `pv` 为空”（即测试 6 的语义），**不**增加依赖机器速度的“恰好在 depth 2 中止”断言。
6. depth 1 前立即中止时，fallback 合法、`score == None`、`completed_depth == 0`、`pv` 为空。
7. PV tracking **不增加任何搜索节点**：见 §8 基线断言（PV 只记录结果，不调用 `try_enter_node`，理论上节点数逐字节相同；有基线才能真正锁住）。
8. UCI 集成：每个完成迭代的 `info` 都有非空 `pv`；最后 `bestmove` 与最后完整 `info pv` 的首着一致。
9. quiescence / qply emergency 中止后**局面完整恢复**；内部临时 PV 的内容不作约束——中止迭代在递归中层层返回 `None` 时，内部 `PvTable` 可以保留部分 PV，这是允许的。硬性不变量只是：① board 必须恢复；② `None` 必须一路向上传播；③ 该 `PvTable` 必须被 `root_search` 整体丢弃；④ 部分 PV **不得**进入 `completed`、`SearchOutcome` 或 UCI `info`。**不要**测试 `assert!(pv.lines.iter().all(Vec::is_empty))`，也**不要**为了清空内部 table 而在每一层 abort unwind 时做额外工作（可复用 `tests/m2_1.rs` 的中断恢复测试模式 + §3.3 的 `debug_assert`）。

### 5.1 测试 2 的断言结构（两不变量分离）

```rust
let before = to_fen(&root);
let outcome = search_best_move(&mut root, ...);
assert_eq!(to_fen(&root), before);          // 不变量 A：搜索对象字节级恢复

let mut replay = root.clone();
for pv_move in outcome.pv {
    assert!(generate_legal_moves(&mut replay).contains(&pv_move));  // 不变量 B：PV 可逐步合法执行
    replay.make_move(pv_move);
}
```

注意：在 clone 上播放 PV 后根局面恢复，那只是因为原对象没被改——不要把"在 clone 上播放 PV 后根局面恢复"当作一个不变量；搜索对象恢复（A）与 PV 可逐步合法执行（B）是两个独立的检查。

   > "for the same promoted piece, a capturing promotion ranks above a quiet promotion"
   （key `(2, promoted_piece_value, victim_value)` 只保证同升变子时吃子升变高于安静升变，不保证任意吃子升变高于任意安静升变，例如 `(2,900,0)` 安静升后仍优先于 `(2,320,900)` 吃子升马。）
2. `tests/m2_2.rs` 的 `ordering_node_count_smoke_bound` 注释：删除 "ordering disabled 会 blow far past 上限" 的推论（无基线支持），只保留"该小局面搜索无异常级节点爆炸"。
