### The three key numbers this milestone establishes

1. **Fixed-depth node count (search-tree efficiency).** Example: startpos
   depth 4 = **8453 nodes** disabled. The heavy middlegames
   (open-tactical 170k, exposed-king / halfmove-ctx 124k) are the
   expensive shapes; low-branch endgames are tiny.
2. **Fixed-node NPS (per-node throughput).** See Throughput below
   (~186k nps at startpos, ~168k open-tactical, ~113-138k queen-win).
3. **Cold / warm TT difference.** A warmed TT collapses node count on an
   already-searched position: startpos 8453 → warm **80** (~1.0%),
   queen-win 2686 → 120 (~4.5%), open-tactical 170068 → 152
   (~0.09%), endgames 6459/29814/9160 → 80/130/85. This is the
   persistent-TT reuse gain, isolated from first-search (cold) cost.

## Throughput (fixed nodes = 100000, disabled, repeat 3)

Three non-terminal, normally-branching fixtures. Node budget is honored
exactly (every run reports `nodes=100000`). `stopped=true` is
expected (search halts mid-iteration at the budget) and is **not** a
failure. `completed_depth` / `bestmove` under a node limit are **not**
treated as a correctness baseline (see spec §10.3).

| fixture | completed_depth | score | bestmove | median NPS (of 3 repeats) |
|---|---|---|---|---:|
| startpos | 5 | cp:40 | b1c3 | 185708 |
| open-tactical | 3 | cp:35 | b1c3 | 167680 |
| queen-win | 6 | cp:900 | e4a4 | 113101 |

`bench_summary suite=throughput mode=disabled fixture_count=3 measured_run_count=9
aggregate_nodes=900000 median_elapsed_us=532624 median_nps=167680`

Queen-win is the slowest per-node (NPS ~113-138k vs ~186k at startpos):
it is a queen-vs-queen+pawns position with many captures, so each
node does more work (more captures to generate/evaluate). This is a
machine/position observation, not a baseline claim.

All three repeats were **identical** (same nodes, score, depth, bestmove,
PV) → the harness printed no `bench_error`, confirming the benchmark is
deterministic.

## Default test-suite runtime (M4.0 added only fast unit tests)

The benchmark module adds unit tests that cover argument parsing,
result-line formatting, `median`, `score` formatting, fixture-ID
uniqueness, FEN parseability, and **one** depth-1 tiny search
(bestmove/PV legality + position restoration). No full suite, no
playout, no wall-clock assertion is in the default gate.

```text
cargo test          (debug,   includes build)  ~26 s wall
cargo test --release (release, includes build)  ~ 3 s wall
```

The 10 added `bench` unit tests add well under one second of actual
test execution (depth-1/2 searches on startpos + pure parsing/
formatting checks). The wall-time totals above are dominated by
compilation, not by the new tests.

## Known limits / P2

- **Single machine, single run.** Numbers are reproducible on this host
  but are not a statistically averaged cloud result. They are a *reference*,
  not a guarantee of cross-machine ratios.
- **Fixture history is single-key** `[pos.zobrist_key()]` by design (see
  spec §7.2). The halfmove component of the TT key is exercised
  via the fixture FEN (e.g. `halfmove-ctx` has `halfmove=80`); true
  *repetition* context (prior distinct positions) is not built into the
  fixtures. This is sufficient for a baseline and keeps it deterministic.
- **No Elo / no speed-percentage claim.** M4.0 establishes a baseline.
  It does **not** assert any performance improvement or playing-strength
  change — none was made.

No sampling profiler was available or runnable in this environment:

- **Windows:** Visual Studio Profiler / WPR+WPA are not installed in
  this sandbox and cannot be launched head-less.
- **Linux / macOS** equivalents (perf, cargo-flamegraph, Instruments)

Per spec §13, rather than fabricate percentages, the missing profile is
recorded as **P2** and does **not** block the benchmark harness
closeout. **No optimization was started because of it.**

Source-level hotspot hypotheses (to be confirmed by a real profiler in a
later milestone, not acted on now):
