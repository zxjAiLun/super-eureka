## 1. Commands run (gate — this measurement campaign)

```bash
# Smoke lock (default reference == M4.0 baseline)
cargo run --release -- bench smoke
cargo run --release -- bench smoke --profile reference
cargo run --release -- bench smoke --profile current

# Standard suite, TT disabled, Reference vs Current
cargo run --release -- bench standard --mode disabled --profile reference
cargo run --release -- bench standard --mode disabled --profile current

# Throughput, fixed 100k nodes, repeat 3, Reference vs Current
cargo run --release -- bench throughput --mode disabled --profile reference --nodes 100000 --repeat 3
cargo run --release -- bench throughput --mode disabled --profile current  --nodes 100000 --repeat 3

# Warm-TT, recorded-only (see Appendix A)
cargo run --release -- bench standard --mode warm --profile reference --repeat 1
cargo run --release -- bench standard --mode warm --profile current  --repeat 1

# Correctness gate
cargo run --release -- perft 5                                        # must be 4,865,609
```

The smoke, standard (§3), and throughput (§4) suites use `--mode disabled`
(TT disabled) so node counts isolate the killer/history ordering effect and are
not dominated by TT reuse. The `--mode warm` runs (Appendix A) use a warmed
TT and are treated as *recorded-only* per spec §7. The fixed-node throughput
pair is run **separately** for `reference` and `current` and compared
fixture-by-fixture on **median NPS**.

> Note: the `current` profile is a real, separate search entry
> (`SearchProfile::Current`). `bench smoke --profile current` therefore runs
> `Current` and is **not** forced to the `1149`/`963` exact lock — those
> locks belong to `reference` only (see §2 and the P1 hardening in §9).

### 4.1 NPS gate (§7: fixed-node NPS drop ≤ 15%)

- Worst per-fixture NPS drop: **-13.5%** (open-tactical) — within ≤ 15%.
- Overall NPS drop: **-13.5%** (median across 9 measured runs per profile = 18 total profile-specific runs) — within ≤ 15%.
- **Outlier robustness:** open-tactical repeat-3 was a ~3× slower outlier
  in *both* profiles (machine contention during that repeat); the median
  correctly excludes it, so it does not distort the comparison.
- **Exact node budget:** each profile's 9 fixed-node runs (18 total
  profile-specific runs) hit the budget exactly (`nodes=100000`,
  `stopped=true`). A node-budget violation is a **fatal validation error**:
  `validate` returns `Err`, `run_one` aborts, the offending `bench_result`
  is never printed, it is excluded from `bench_summary`, and the CLI exits
  non-zero (P1 #2 hardening `5693b190ab3978e012c592e5cf23174b4802f371`, §9).
- **P2.1 arithmetic:** the original review corrected the prior report's
  `-4.9%` to `-5.2%` (from the original run's 188,869→179,054).
  This re-measurement produced different NPS (machine variance, above), so
  the real drop for *this* run is `-13.5%`; `-5.2%` remains the
  correct arithmetic for the original run's numbers.

> The node reduction (52.6%) overwhelmingly pays for the per-node
> killer/history bookkeeping cost. Throughput (fixed-node) wall time is
> intentionally node-bound and similar; the speed win shows up in the
> fixed-depth standard suite (§3 / §6).

**None.**

## 9. What changed (scope)

### Commit 5 — `a4224a8` `bench: record M4.1 search deltas`

- `src/engine/bench.rs`: added an **optional** `--profile reference|current`
  flag (default `reference`, so every existing M4.0 command reproduces
  exactly). `reference` drives the original M4.0 entry; `current` drives the
  profile-aware entry with `SearchProfile::Current`. `bench_result` and
  `bench_summary` now emit `profile=`. No search semantics changed.
- `README.md`: added a short `--profile` usage note.
- `src/engine/search.rs` and `src/uci.rs`: **not modified** (no production
  search-semantics change). The retained M4.0 entry
  (`search_best_move_with_history_and_tt`) stays as the `reference` path and
  is still exercised by the M4.0 reference tests.
- `docs/benchmarks/m4.1-quiet-move-ordering.md`: this report.

### P1 hardening #1 (pushed `7ade86e`) — `fix(bench): scope smoke locks to reference profile`

- `bench.rs` `validate()` now receives the `SearchProfile` and the exact
  smoke lock is applied **only** when
  `mode == BenchMode::Disabled && profile == SearchProfile::M4Reference`.
  `run_one()` passes `cfg.profile` into `validate()`.
- `bench smoke --profile current` therefore runs `Current` to completion
  without forcing the `1149`/`963` exact lock; `bench smoke` and
  `bench smoke --profile reference` still enforce it exactly.
- Added quick tests: default/smoke profile is `M4Reference`; reference smoke
  still locks exactly; current smoke completes both fixtures and emits
  `profile=current`; a direct `validate` contract test proving `Current`
  ignores a deliberately-wrong lock while `M4Reference` enforces it.
- `7ade86e` also added the node-budget invariant in `validate()` (a measured
  `Nodes(n)` run must satisfy `nodes <= n` / `stopped == true` / `nodes == n`),
  but initially only emitted a non-fatal `bench_error` warning. That was
  rejected as P1 in the next review (a budget violation must be fatal).
- Comment cleanup: module doc now accurately states `reference` drives the
  M4.0 entry and `current` drives the profile-aware entry; the harness
  itself never alters search semantics.

### P1 hardening #2 (`5693b190ab3978e012c592e5cf23174b4802f371` `fix(bench): reject invalid node-budget results`) — fatal node-budget validation

- The node-budget invariant in `validate()` is now a **fatal validation
  error**. On `LimitKind::Nodes(n)` it returns `Err` when
  `nodes > n`, or `!outcome.stopped`, or `nodes != n`. Consequently
  `run_one` propagates the error, the offending `bench_result` is never
  printed, it is excluded from `bench_summary`, and the CLI exits non-zero
  with a `bench_error` line. This matches the fatal behaviour of the
  fixed-depth check rather than the soft determinism warning.
- Added a negative unit test (`node_budget_invalid_results_rejected`): a real
  search outcome is fed to `validate` with (a) `nodes` one short of the
  budget (under-budget) and (b) a hand-built `SearchOutcome` with
  `stopped = false`. Both must return `Err`, proving a budget violation is
  rejected rather than silently passed. The existing exact-hit test
  (`node_budget_measured_hits_exact`, 50 nodes) is retained.

> Note: warm-TT mode remains available but is **recorded-only** per spec §7
> (TT reuse dominates there and is not the primary proof of the killer/history
> effect). See Appendix A.
