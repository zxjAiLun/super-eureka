### 2.3 Single-fixture node-increase check (§spec: >20% must explain, >30% blocks)

**None exceed 20%.** The small increases are:

| fixture     | m4.1 | current | Δ     | change | cause |
|-------------|------|---------|-------|--------|-------|
| queen-win   | 1,213| 1,310   | +97   | +8.0%  | scout(s) improved alpha and therefore required a full-window re-search; the re-search re-walks the subtree |
| high-branch | 2,818| 3,014   | +196  | +7.0%  | shallow depth-3 wide tree; some scouts improved alpha and were fully re-searched |
| rook-pawn   | 5,205| 5,455   | +250  | +4.8%  | low-branching endgame; a few alpha-improving scouts required re-search |
| krk         | 9,064| 9,065   | +1    | +0.01% | single extra node from PVS bookkeeping; negligible |

All are **below 8%** and are the expected cost of PVS re-searches. The
precise re-search semantics per site are:

```
non-root, alpha < scout_score < beta : scout improves alpha → full-window
                                       re-search (this is the only non-root
                                       re-search case)
non-root, scout_score >= beta        : direct fail-high cutoff — NO
                                       re-search
root (no beta cutoff at the root)    : any later scout that improves alpha
                                       → full-window re-search
```

The extra nodes above can only come from "scout followed by a required
full-window re-search". The harness does not trace re-search sites, so this
report deliberately does **not** attribute the increases to root vs
non-root re-searches or claim a specific re-search count — that attribution
would require a trace the harness does not produce. `kqk` is exactly
**0.0%** (29,762 = 29,762): the first root move is best and every later
scout fails low (`scout_score <= alpha`, no re-search), so PVS adds zero
nodes. No fixture approaches the 20% explain threshold or the 30% block
threshold.

  order reversed, shows Current *faster* overall (−5.0%).

These observations may reflect machine-state variance (core-frequency
boost, contention), may include a real per-node profile cost difference, or
both — the available data cannot distinguish these, and this report does
not attempt to. Even at an identical node budget the two profiles can visit
different position mixes, produce different qsearch fractions, have
different TT-hit and re-search behavior, and perform different amounts of
ordering, draw-check, and PV work, so a genuine per-node cost difference is
possible in principle. The NPS gate therefore stands **FAIL / UNRESOLVED**
pending independent remote review and direction.
