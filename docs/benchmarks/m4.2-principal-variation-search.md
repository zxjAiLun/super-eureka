## 5. Fixed-node throughput (disabled, 100k nodes, repeat 3) — NPS gate

`nodes` budget is always exactly 100,000 (atomic node-budget semantics;
verified in every run: `nodes=100000`, `stopped=true`, no `bench_error`).
Compared on **median NPS**.

### 5.1 Campaign used for the gate

An initial throughput run produced an **invalid median comparison**: the
`current` profile happened to catch **two** slow `open-tactical` repeats
(109,190 / 61,984 NPS) while `m4.1` caught only **one** (61,802 NPS). With
repeat-3, a single outlier is excluded by the median (as documented in the
M4.1 report §4.1), but two outliers *inside* the median make the comparison
meaningless — it reported a 24.0% overall median NPS drop that is physically
implausible for PVS (a null-window search walks the same tree as a
full-window search; per-node cost is near-identical). A re-run was therefore
performed in which **both** profiles experienced the same reproducible
`open-tactical` contention pattern (one ~3× outlier each). The re-run's
median is a valid comparison and is the gate evidence below. The initial
run's raw data is preserved in §5.3 for transparency; nothing is filtered or
beautified.

### 5.2 Gate result (re-run, valid median)

Per-fixture median NPS (median of 3 repeats):

| fixture       | m4.1 median NPS | current median NPS | Δ NPS   |
|---------------|------------------|--------------------|---------|
| startpos      | 217,015          | 196,098            | **-9.6%** (worst) |
| open-tactical | 129,160          | 121,974            | -5.6%   |
| queen-win     | 99,203           | 105,880            | +6.7%   |

Overall median NPS = median of all 9 measured runs per profile:

```
M41Reference overall median NPS : 129,160
Current      overall median NPS : 121,974
Overall median NPS drop         : 5.6%
Gate: overall median NPS drop ≤ 10%  →  PASS
Worst per-fixture drop           : -9.6% (startpos)  →  within ≤ 10%
```

All 18 fixed-node runs (9 per profile) hit the budget exactly:
`nodes=100000`, `stopped=true`, no `bench_error`. A node-budget violation is
a fatal validation error (P1 #2 hardening `5693b19`, see M4.1 report §9):
`validate` returns `Err`, `run_one` aborts, the offending `bench_result` is
never printed, and the CLI exits non-zero. No such violation occurred.

The ~5.6% per-node overhead is the expected cost of PVS bookkeeping: the
scout-vs-full branch decision and the occasional full re-search after a
scout fail-high. On `queen-win`, Current is actually **faster** per node
(+6.7%) — within run-to-run noise. This confirms PVS is per-node
near-neutral, and the M4.2 win shows up as a node-count reduction (§2), not
as a per-node slowdown.

## 9. No Elo conclusion

This report makes **no Elo claim**. The bench harness measures deterministic
search-tree size (nodes), fixed-depth wall time, and fixed-node throughput on
a fixed fixture set. It does not play games, does not estimate playing
strength, and does not compare against external engines. Any Elo implication
would require a separate game-play benchmark, which is explicitly out of
scope for M4.2.
